# Update History Standard: Gravity Freight V2

## 1. 目的
`update_history.json` は、プロジェクトの進化を「ユーザー体験（機能・品質）」の観点から記録し、ユーザーが最新版へのアップデートを検討する際の判断材料を提供するためのものである。
したがって、開発内部の技術的な詳細、リファクタリング、ライブラリの更新、テストの追加といった「ユーザー体験に直接寄与しない内部記録」は、記載の対象外とする。

## 2. 記述の基本原則
- **ユーザー体験優先**: プレイヤーや運用者に直接影響のある「追加機能」「UI改善」「不具合修正」のみを記載する。
- **内部作業の排除**: リファクタリング、ファイル分割、テスト作成、ライブラリ導入などの「内部作業ログ」は記載しない。
- **簡潔な記述**: 専門用語を避け、変更によって何が可能になったか、何が改善されたかを簡潔に記す。

## 3. 記述フォーマット
具体的な JSON 構造については、[update_history.json 仕様書](../specs/common/update_history.md) を参照すること。
履歴を追加する際は、常に**配列の先頭**に追加し、最新の更新が最初に表示されるようにする。

### 記載項目の分類
JSON の `changes` 配列内では、以下の分類を意識して記載する。

- **機能追加 (Feature)**: 新しく追加された機能やコンテンツ。
- **機能改善 (Enhancement)**: 既存機能のブラッシュアップ、UI/UXの向上、パフォーマンス改善、データ整合性の強化など。
- **不具合修正 (Bugfix)**: ユーザーに影響のあった挙動の修正。
