# Coding Principles & Review Standard: Gravity Freight V2

本ドキュメントは、プログラミングの本質を捉えた「7つの設計原理」を Gravity Freight V2 の開発に適用し、持続可能なコード品質を維持するための基準を定義する。

---

## 1. 開発の 7 つの設計原理 (7 Design Principles)

### 1-1. 単純性 (Simplicity)
**「複雑さを最小限に抑える」**
設計において「単純であること」は最も重要な指針である。余計な要素を削ぎ落とし、誰が見ても理解できる状態を保つ。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **500ステップ制限**: 1ファイルあたりの行数が500行以内に収まっているか？
    - [ ] **詳細度の抑制**: CSS で無意味な ID 指定や、過剰に深いセレクタネストを排除しているか？
    - [ ] **解決策の選択**: 複数の方法がある場合、トリッキーなコードではなく最も単純な方を選んでいるか？

### 1-2. 同型性 (Isomorphism)
**「似たものは同じ形にする」**
同じ役割、同じ意味を持つものは、一貫した形式（命名、構造、スタイル）を持たせることで、直感的な理解を助ける。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **コンポーネントの一貫性**: カード、バッジ、ボタン等の構造が全画面で統一されているか？
    - [ ] **命名規則の統一**: 同じ機能を持つ関数やプロパティに、異なる名称（例：`show` と `display`）を混在させていないか？

### 1-3. 対称性 (Symmetry)
**「一貫性と対称性を保つ」**
機能や概念の対（ペア）になる部分は、実装の作法も対称にすることで、パターンの予測を可能にする。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **対機能の実装**: 追加と削除、表示と非表示などの対になる処理が、同じ論理構造で書かれているか？
    - [ ] **デザインの対称性**: `Neon`（発光）と `Matte`（非発光）など、異なるスタイルの適用方法が対称的であるか？

### 1-4. 階層性 (Hierarchy)
**「抽象化のレベルを整理する」**
要素間の関係を階層として整理し、各レイヤーが適切な抽象度で責務を果たすようにする。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **Axis設計の遵守**: Base（基盤）、Layout（構造）、Style（装飾）、Context（文脈）のレイヤーが明確に分かれているか？
    - [ ] **責務の分離**: 上位レイヤーが下位レイヤーの具体的な詳細（固定ピクセル値など）を不適切に肩代わりしていないか？

### 1-5. 線形性 (Linearity)
**「関係を単純な直線にする」**
機能や状態の因果関係を単純な直線（Aを変更すればBが変わる）にし、複雑な絡み合い（スパゲッティ状態）を避ける。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **副作用の排除**: JS において、グローバル変数の書き換えや、予期せぬ場所への副作用を排除しているか？
    - [ ] **CSSの独立性**: 1つの CSS を変更した際に、無関係な箇所のレイアウトが壊れるような複雑な依存関係がないか？

### 1-6. 明晰性 (Explicitness)
**「意図を明確にする」**
コードや設計において「何をしているか」が隠蔽されず、明示的で分かりやすい状態にする。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **マジックナンバーの排除**: 数値や文字列に意味のある定数名を付け、意図を明確にしているか？
    - [ ] **ロジックの透明性**: `!important` や `z-index` の場当たり的な使用など、不透明な解決策を避けているか？

### 1-7. 安全性 (Safety)
**「不具合を最速で顕在化させ、最終的な品質を担保する」**
開発時における安全性とは、異常を沈黙させずに即座に停止させ、問題を早期に発見できる状態を指す。バグを隠蔽して無理に動かし続けることは、将来の致命的な欠陥を招く「不安全な行為」である。

- **V2 レビュー基準 (Checklist)**:
    - [ ] **不具合の自然な顕在化 (Natural Failure)**: 欠落したプロパティや異常な値に対して過剰なチェック（if 文の乱立など）を行わず、NaN や undefined、表示の崩れとして自然に不具合が顕在化することを許容しているか？（無理にデフォルト値や try-catch で保護して「動いているふり」をさせない）
    - [ ] **バグ隠蔽の禁止 (No Defensive Code)**: 「念のため」の try-catch や null チェック、デフォルト値へのフォールバックなど、不具合を隠蔽し得る防御的プログラミングを排除しているか？
    - [ ] **TDDの実施**: 報酬計算や経済バランスなどの重要ロジックに対し、事前にテストコードで安全性を担保しているか？
