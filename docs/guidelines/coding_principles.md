# Coding Principles & Review Standard: Gravity Freight V2

本ドキュメントは、ソフトウェアエンジニアリングの基本原則である「7つの原理」を Gravity Freight V2 の開発に適用し、持続可能なコード品質を維持するための基準を定義する。

---

## 1. 開発の基本原理 (Core Principles)

Gravity Freight プロジェクトにおけるすべての実装は、以下の原理に照らして審査される。

### 1-1. 単純原理 (Keep it Simple)
**「複雑さを最小限に抑える」**
- 無意味な詳細度（ID指定の多用）や、深いネストを排除する。
- 物理的なボリュームを抑制する。1ファイルあたりのステップ数（行数）を **500ステップ以内** に抑えることを目安とし、それを超える場合は「階層原理」に基づいてリファクタリング（ファイル分割や抽象化）を検討する。
- **コメントの最小化**: コードを読めばわかる内容（What）をコメントに書くのはノイズであり、単純原理に反する。コメントが必要なほど複雑なコードは、コメントで説明する前にまず構造の単純化を検討する。
- 解決策が複数ある場合、最も単純で理解しやすい方を選択する。

### 1-2. 同型原理 (Isomorphism)
**「似たものは同じ形にする」**
- 役割が同じコンポーネント（カード、バッジ等）は、命名・構造・スタイル適用方法を一貫させる。
- 「例外」を作らず、パターンを統一して認知負荷を下げる。

### 1-3. 対象原理 (Symmetry)
**「一貫性と対称性を保つ」**
- 機能の対（追加と削除、表示と非表示）やスタイル（Neon と Matte）の間で、実装の作法を対称にする。
- プロパティの指定方法や変数の利用方法が「片方だけ特殊」という状態を許容しない。

### 1-4. 階層原理 (Hierarchy)
**「抽象化のレベルを整理する」**
- Base（基盤）、Layout（構造）、Style（装飾）、Context（個別文脈）のレイヤーを明確に分ける（Axis設計）。
- 上位レイヤーが下位レイヤーの責任（共通の高さ指定等）を冗長に肩代わりしない。

### 1-5. 線形原理 (Linearity)
**「関係を単純な直線にする」**
- 副作用を避け、Aを変更すればBが変わるという因果関係を明確にする。
- 複雑にリンクし合った CSS や、グローバルな状態を書き換える JS を排除する。

### 1-6. 透明原理 (Transparency)
**「隠れた意図を排除する」**
- コードを読んだだけで、副作用や挙動が予測できるように記述する。
- 内部状態の見えない「魔法の定数」や、暗黙的なフォールバック（不要な default 指定）を排除する。

### 1-7. 証明原理 (Proof / Verifiability)
**「正しさを論理的に説明できる」**
- 「なんとなく動く」ではなく、なぜその変数が、なぜその場所で定義されているかを論理的に説明可能なコードにする。
- 冗長な記述がないことが、コードの純粋性によって証明されている状態を目指す。

### 1-8. 安全原理 (Safety)
**「異常を沈黙させない」**
- 「プログラムが停止しない（クラッシュしない）」ことは、開発期間における安全ではない。
- 異常が発生した際に、それを包み隠さずエラーとして即座に表面化させることこそが真の安全である。
- 「念のため」のフォールバック指定や例外の握り潰しは、バグを潜在化させる不安全な行為として排除する。「念のため」を削ることは、不具合を確実に検知するための安全装置である。

### 1-9. 透明原理（コメント）：Why, not What
**「コードの意図を透かす」**
- コメントは「何をしているか」ではなく、「なぜそうしたか（背景、制約、意図）」を記述するために存在する。
- ファイルヘッダーや主要関数に JSDoc 形式で論理的なインターフェース（役割、引数、戻り値）を明示し、詳細な実装を追わずとも役割が理解できる「透明性」を確保する。

## 2. 実践的チェックリスト

### 2-1. CSS: 構造と美学の健全性
- [ ] **同型性**: 似た役割のバッジやボタンが、全く異なるクラス名やプロパティ構成になっていないか？
- [ ] **階層性**: `play.css`（Context）で、`ui_layout.css`（Base/Layout）の共通プロパティを物理的な数値で無意味に上書きしていないか？
- [ ] **透明性**: `var(--current-color)` のフォールバックのように、定義が追いづらくなるマジックを排除しているか？
- [ ] **単純性**: セレクタを簡潔に保っているか？（詳細度競争になっていないか？）

### 2-2. JS: 論理と状態の健全性
- [ ] **線形性**: 関数が予期せぬグローバル変数を書き換えていないか？（純粋関数の維持）
- [ ] **証明性**: その関数の上書きや変数の再初期化は、論理的に「今、そこで」行われるべき必然性があるか？
- [ ] **安全性**: 誤った引数や状態に対して、適切な保護やセーフティネット（デザインデフォルト値など）が機能しているか？

### 2-3. ドキュメントとコメント
- [ ] **コードの二重説明になっていないか**: 「AをBに代入する」といった、コードをなぞるだけのコメントはないか？
- [ ] **「なぜ」が書かれているか**: 特殊なマジックナンバーや非直感的なロジックに対して、その背景や意図（Why）が記述されているか？
- [ ] **JSDoc の正確性**: 引数や戻り値の型・役割が、実装と乖離していないか？

### 2-4. 重複の排除 (DRY vs Simple)
- [ ] 同じ値が3回以上登場したら「階層原理」に基づき、抽象化された Base クラスや Tokens へ引き上げる検討をしたか？

## 3. レビュー時の心構え

- **「なぜ冗長なのか」を原理で説く**: 単に「消して」ではなく、「階層原理に反している（上位で定義済み）」「同型原理に反している（他と書き方が違う）」と原理に基づいて議論する。
- **美しさは正しさ**: 透明性が高く、単純なコードこそが最も美しい。その「美徳」をプロジェクトの共通認識とする。
