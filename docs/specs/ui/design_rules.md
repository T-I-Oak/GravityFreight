# GRAVITY FREIGHT: ORBITAL ECHOES - 仕様書 (UI基本デザイン)

## 1. 画面（シーン）の種類
以下の画面（シーン）構成とする。

- メニュー画面
    - ゲーム開始時に表示される画面で、他の画面への入り口となる画面
- プレイ画面
    - 実際にゲームをプレイする画面
- 航行結果画面
    - 1回の航行ごとに表示される、航行結果（獲得コイン、獲得パーツ、到達距離など）を表示する画面
- 施設画面
    - セクターをクリアーしたとき、出口（アーク）に応じて発生するイベントの画面
- ゲームリザルト画面
    - ゲーム終了時に表示される、トータルスコアーやグレードを表示する画面
- チュートリアル画面
    - ゲームの遊び方を表示する画面
- 記録画面
    - ランキング、アチーブメントなどを表示する画面

## 2. 画面スタイル
ゲーム全体で、3種類の画面スタイルを持つ。
- ネオンスタイル
    - 宇宙空間を表すスタイル。ネオン発光とグラスモーフィズムによる透明感が特徴。
- マットスタイル
    - 施設内を表すスタイル。派手すぎないが、ややポップな雰囲気が特徴。
- 印刷スタイル
    - 事務的な通知を表すスタイル。紙に印刷された雰囲気が特徴。

## 3. 色
### 3.1 基本カラー
- メイン背景色: `#050510` (深い宇宙の闇を表現するベース色)
- メインカラー: `#00ffcc` (高エネルギーの反応を示すシアン)
- アクセントカラー: `#ff6600` (警告や重要導線を示すオレンジ)
- サブカラー: `#4488ff` (補助的なインターフェースを示すブルー)

### 3.4 動的カラー生成ルール (Dynamic Color System)
JSから渡される単一の基本色 `--ui-color` から、コンポーネントの質感を自動生成する。

- **Surface (背景)**: `color-mix(in srgb, var(--ui-color), transparent 85%)` (15%着色を標準とする)
- **Border (枠線)**: `color-mix(in srgb, var(--ui-color), white 20%)` (発光感を出すため少し明るく)
- **Glow (発光)**: `var(--ui-color)` をベースとした `box-shadow` / `text-shadow`
- **Text (文字)**: 基本的に `var(--ui-color)` または `white` (視認性優先)

## 4. デザインルール
UIの「形（レイアウト/形状/情報構造）」は全スタイルで共通化し、「見た目（色/質感/発光/透明度）」はスタイル層で差し替える。
これにより、共通定義を変更した際に、全スタイルの形状が必ず同期して変化することを担保する。

### 4.1 階層（共通定義とスタイル定義の分離）
UIの見た目は、必ず次の2階層で実装できる形にする。

- 共通定義（Base）
    - 形状: 角丸、余白、レイアウト、タイポグラフィの基準、情報の並び順
    - インタラクション: hover/active/disabled 等の状態遷移（ただし「色や光り方」は含めない）
    - コンポーネントの内部構造（例: アイテムカードの「左ストライプ」「アイコン」「名称」「スペック」など）
- スタイル定義（Style Layer）
    - 色、透明度、影/発光、ノイズ/紙感などの質感
    - 枠線の太さは原則スタイル側（ただし当たり判定やレイアウトに影響する寸法は Base）

#### 4.1.1 Heroスケールにおける例外（ウィンドウサイズ依存のスケーリング）
タイトル画面など、ブランドの視覚的アイデンティティを維持する必要があるHero画面においては、デザインの黄金比を担保するために、ウィンドウサイズに基づいた動的スケーリング（vw 単位と clamp() の併用）を必須とする。これにより、HUD（プレイ画面）の基準値調整から独立しつつ、あらゆる解像度で意図したバランスで表示されることを担保する。

### 4.2 スタイルの適用単位（画面ルート）
画面（シーン）ルート要素にスタイル識別子を持たせ、スタイル差分はその配下でのみ適用する。

- ネオンスタイル: ui-style--neon
- マットスタイル: ui-style--matte
- 印刷スタイル: ui-style--print

スタイル差分は「共通コンポーネントのクラス」を上書きする形で定義する（スタイルごとに別クラスへ分岐させない）。

### 4.3 状態（state）と役割（role）の命名
コンポーネントの役割クラス（Base）と状態クラス（state）を分離する。

- 役割（Base）: `ui-button`, `item-card` など
- 状態（State）: `is-clickable`, `is-disabled`, `is-selected` など

状態クラスは見た目を直接固定せず、スタイル層で表現方法（明るくする/発光する/インクが薄くなる等）を差し替えられるようにする。

### 4.4 コンポーネント別の共通定義（Base）
共通定義は「形が壊れない」ことを最優先とし、色や質感はスタイル層へ追い出す。

#### ボタン（共通）
- 角ボタン: radius 4px
- 丸ボタン: radius 50%
- hover/active の変化は「存在する」ことだけを共通で保証し、表現（色/発光/陰影）はスタイル層に委ねる

#### disabled（共通）
- 操作不能であることが視覚的に分かること
- 表現（破線/薄さ/印字かすれ等）はスタイル層に委ねる

#### clickable（共通）
- クリック可能であることが視覚的に分かること
- hover 時に反応があること（例: 右に 2px ずらす等、ただし最終表現はスタイル層で調整可）

#### アイテムカード（共通）
アイテムカードは「形状と内容」は共通とし、「質感」はスタイルで差し替える。

- 形状（Base）
    - radius 4px
    - 左ストライプ（カテゴリ色）を持つ
    - 情報構造（標準カード）
        - 1行目: アイテム名 / スタック数 / 耐久度
        - 2行目: 説明文
        - 3行目以降: プロパティ（横並び、または折り返し）
    - 密度によって表示行を削る場合でも「削ってよい領域」を固定する
        - `density-compact` では説明文を表示しない（名称/主要スペックの可読性を優先）
        - スタック数/耐久度/主要プロパティは省略しない
    - ロケットカード（特殊: ROCKET）
        - 1行目: `[シャーシ名] - [ロジック名]`
        - 2行目: プロパティ（横並び、または折り返し）
        - 3行目以降: 搭載モジュール一覧（`[モジュール名] / [耐久度]`）
- カテゴリ色（Base）
    - 左ストライプの色は「アイテムのイメージカラー」を使用する
    - それ以外の適用（発光させる/背景に滲ませる等）はスタイル層で決める

#### タイトルボックス（.ui-title-box）
- タイトル文字（.text-display）とサブタイトル（.text-sub-display）をセットにした共通ロゴブロック。
- 特徴:
    - 内部は常に中央揃え（align-items: center）。
    - タイトル文字は `ui--style-neon` 時、画面全体のフォントサイズが変動しても `var(--text-display-size)` で固定されるよう強制する。
    - サブタイトルは画面幅に応じてレスポンシブに縮小し、改行を禁止する。

#### ステータスボックス（.ui-stat-box）
- ラベル（.stat-label）と数値（.stat-value）をセットにした共通の計器表示ブロック。
- 特徴:
    - 垂直配置（Label:上、Value:下）を基本とする。
    - 数値部分には用途に応じたセマンティッククラス（.num-sector など）を併用する。

### 4.5 スタイル別の差分（Style Layer）
各スタイルは、Base コンポーネントの見た目を「上書き」することで差分を作る。
このとき、カードやボタンのレイアウト寸法（padding, gap, font-size 等）を勝手に変えない。

- ネオンスタイル
    - 半透明、発光、グラスモーフィズムを許可
- マットスタイル
    - 不透明寄り、陰影は控えめ、読みやすさ重視
- 印刷スタイル
    - 紙/インクの雰囲気、強い発光は禁止

### 4.6 実装ルール（CSS）
スタイル階層を安定して運用するため、以下を必須とする。

- `!important` は使用しない
    - 例外は作らない（運用上の例外は必ず負債化するため）
- 色はハードコーディングしない
    - 直接のカラーコード指定（`#...`, `rgb(...)` 等）を散在させない
    - 色は CSS 変数（例: `var(--color-...)`）を参照する

#### 4.6.1 文脈によるスタイル適用 (Contextual Styling)
UI実装者が「質感」を意識しなくて済むよう、親要素（画面ルートやパネルルート）がその配下にある全コンポーネントのスタイルを決定する。

- **ルール**:
    - 個別コンポーネントに `.btn-neon` や `.btn-matte` といった質感クラスを直接付与しない。
    - コンポーネントには `.ui-button` や `.ui-panel` といった「機能的ベースクラス」のみを付与する。
    - 親要素（`<body>` や `#title-screen` 等）に `.ui-style--neon` 等のコンテキストクラスを付与する。
- **実装イメージ**:
    - CSS側で `.ui-style--neon .ui-button { ... }` のように定義することで、実装者は「ボタンを置く」という行為だけで正しい質感を自動的に得ることができる。

### 4.7 選択式トークン（許可された値セット）
UIの統一感を保つため、「個別に任意の値を決めてよい」項目を極力なくす。
具体値が未確定の段階では、まず **選択肢（スロット）だけ** を定義し、実装ではその中から選ぶ。

- 方針
    - 個別コンポーネントが `blur: 5px` のように勝手な値を追加してはいけない
    - まず「small / large」等のスロットを定義し、実値は CSS 変数で一元管理する
    - 変数は `ui-style--...` で差し替え可能にする（同一スロットを全スタイルで共通運用）

#### 4.7.1 視覚効果トークン（Visual Effects）
必要最小限の質感スロット。

- **blur / glow** (質感)
    - `--blur`: `12px` (ネオン時のグラス背景ぼかし)
- **Glow Levels (Text/Object)**:
    - 原則: 「鋭い芯 (Core)」と「広い余韻 (Glow)」の 2〜3 層構造とする。背景に埋もれず、かつネオンの幻想的な質感を維持する。
    - --glow-text-hero: 0 0 10px [Color], 0 0 20px [Color], 0 0 30px [Color] (Heroロゴ専用の特大スケール)
    - --glow-text-high: 0 0 2px [Core], 0 0 8px [Color] (標準的な強調要素)
    - --glow-text-comfort: 0 0 2px [Core], 0 0 3px [Color] (HUD数値などの視覚的疲労軽減モデル)
    - --glow-text-low: 0 0 2px [Color] (補助的な発光)
- **opacity** (不透明度)
    - `--opacity-surface`: `0.6` (ネオン) / `0.95` (マット)
    - `--opacity-overlay`: `0.8` (背面暗幕)
- **border-style**
    - `--border-style-main`: `solid` (通常) / `dashed` (印刷)

#### 4.7.2 レイアウトトークン（Layout）
迷いをなくすための 2 段階の余白と角丸。

- **spacing / gap**
    - `--space-s`: `8px`  (コンテンツ内部、ボタン間)
    - `--space-m`: `16px` (パネルのパディング、セクション間)
- **radius**
    - `--radius-m`: `12px` (標準の角丸)
    - `--radius-round`: `50%` (サークル)
- **border-width**
    - `--border-1`: `1px`
    - `--border-2`: `2px`

### 4.8 レスポンシブ指針（勝手に増やさない）
レスポンシブ対応は、個別画面ごとに独自ブレークポイントを作らない。

- 画面幅の区分は、プロジェクトで定義したセット（例: `sm / md / lg`）のみを使用する
- レイアウト変更の基本は「1カラム化」「情報の折りたたみ」「優先度順の並び替え」
- 文字サイズや余白も、トークン（`--space-*`, `--font-*` 等）経由で調整する

### 4.9 スクロールバー指針（統一）
スクロールバーはコンポーネントごとに独自スタイルを持たせず、プロジェクトで統一する。

- 原則
    - スクロールが必要な領域は、スクロールコンテナの役割クラスを付与して統一管理する
    - スクロールバーの色/太さ/角丸はトークンで定義し、Style Layer で差し替える
- 例（スロット）
    - `--scrollbar-size`
    - `--scrollbar-track`
    - `--scrollbar-thumb`

#### 4.9.1 タイポグラフィ（Typography）
情報のヒエラルキーと世界観を両立させるためのサイズ体系。

- **フォントファミリー**
    - `--font-display`: `'Outfit', sans-serif`
    - `--font-text`: `'Inter', sans-serif`
    - `--font-mono`: `'Courier Prime', monospace`
    - `--font-special`: `'UnifrakturMaguntia', cursive`
    - `--font-num-primary`: `'Orbitron', sans-serif`
- **ウェイト基準 (Neon Texture)**
    - ネオンスタイルにおける「質感」としての標準ウェイトは 900 とする。これにより発光の「芯」を太く保ち、重厚な質感を全画面で共有する。
- **サイズスロット**
    - --text-display-size: clamp(2.5rem, 10vw, 4.5rem)
    - --text-sub-display-size: 1.5rem
    - --text-heading-size: 1.25rem
    - --text-body-size: 14px (ボタンおよび本文の基準サイズ)
    - --text-caption-size: 0.75rem
- **スペーシング**
    - `--text-line-heading`: `1.2`
    - `--text-line-body`: `1.6`
    - `--text-letter-spacing-display`: `0.1em`

#### 4.10.2 ボタン・バリエーション (Variants)
V1の「導線の強弱」を継承する。

- **Primary (Emphasis)**: 
    - 状態: コンテキストに応じた基本色で塗りつぶされる。
- **Secondary (Outline)**:
    - 状態: 控えめな枠線のみ。ホバー時に色が宿る。
- **Context Modifiers**:
    - `.is-repair`, `.is-trade`, `.is-market`: 各施設に応じた成功/前進アクション。
    - `.is-retry`: オレンジ色 (`var(--color-status-gameover)`)。帰還、墜落、消失など、セクターをやり直す際のアクションに使用。

#### 4.9.2 モーション（Motion）
- **速度**
    - `--motion-fast`: `150ms`
    - `--motion-normal`: `300ms`
    - `--motion-slow`: `600ms`
- **イージング**
    - `--ease-standard`: `cubic-bezier(0.4, 0, 0.2, 1)`
    - `--ease-emphasis`: `cubic-bezier(0.34, 1.56, 0.64, 1)` (弾むような強調)

#### 4.9.3 レイヤー（Z-index）
- **階層**
    - `--z-base`: `1`
    - `--z-hud`: `100`
    - `--z-dialog`: `1000`
    - `--z-tooltip`: `2000`

### 4.10 入力ポリシー（装飾キー）
本作はキーボード操作（フォーカス移動やショートカット）をサポートしない。
ただし、マウス操作を補助するための装飾キー（Ctrl / Shift 等）は使用する。

- フォーカス
    - キーボードフォーカスによる操作・フォーカスリング等は対象外とする
- パン（マップ移動）
    - 対象: プレイ画面のマップ（カメラ）のみ
    - `Ctrl`（または `Shift`）+ ドラッグでカメラ（マップ）をパンできる

### 4.11 数値表記と数値スタイル（統一）
数値は表記・見た目を統一し、画面ごとに勝手なフォーマットや強調を作らない。

- 表記
    - 3桁カンマ区切り（例: `1,500`）
    - 数値の種類（スコア/セクター/コイン/アイテム数）ごとに役割スタイルを固定する

#### 単位（表示ルール）
- スコア: `pts`（suffix）
- コイン: `c`（suffix）
- アイテム数: `pcs`（suffix）
- セクター:
    - `scs`（suffix）
        - ゲームオーバー画面等で「端末レポート」風に表示するための表記
        - 例: `2 scs`
    - `sts` は使用しない

- スコア: `num-score` (エネルギー感の強いシアン)
- セクター: `num-sector` (冷静な判断を促すシルバー、発光色はブルー)
- コイン: `num-coin` (価値の象徴であるゴールド)
- アイテム数: `num-item-count` (視認性の高いホワイト)

#### 4.11.3 数値スタイルトークン
- **フォント**
    - `--font-num-primary`: `'Inter', sans-serif`
    - `--font-num-secondary`: `'Inter', sans-serif`
- **色（セマンティック）**
    - `--color-num-score`: `var(--color-primary)`
    - `--color-num-sector`: `var(--color-sub)`
    - `--color-num-coin`: `var(--color-coin)`
    - `--color-num-item-count`: `var(--color-text-main)`

### 4.12 段階・序列の表示（グレード/ランキング/進捗）
GRADE、ランキング上位、アチーブメントの到達段階など「レベル分け」を共通の役割として定義し、画面ごとに勝手な表現を作らない。

- 役割（例）
    - GRADE: `badge-grade`
    - ランキング順位: `badge-rank`
    - アチーブ段階: `badge-tier`
- 段階（スロット）
    - `tier-1`, `tier-2`, `tier-3`, `tier-4`, `tier-5`
    - `rank-1`, `rank-2`, `rank-3`（TOP3 表示用）
    - `rank-top`（TOP3 以外の上位を別表現にする場合）
    - `rank-normal`
    - `rank-low`（例: 21位以降など、圏外表現が必要な場合）
#### 4.12.3 段階別カラー・トークン (Grade/Tier Colors)
評価の直感的な識別を可能にするためのカラーセット。

- **カラー**
    - `--color-tier-1` (S / 1st): `#ff3300` (最高評価 / 警告級レッド)
    - `--color-tier-2` (A / 2nd): `#ffd700` (優秀 / ゴールド)
    - `--color-tier-3` (B / 3rd): `#00ffcc` (良好 / シアン)
    - `--color-tier-4` (C): `#4488ff` (普通 / ブルー)
    - `--color-tier-5` (D/E/F): `#546e7a` (低評価 / 紺・グレー)
- **バッジ形状**
    - `--badge-radius`: `2px`
    - `--badge-padding-x`: `6px`
    - `--badge-padding-y`: `2px`

### 4.13 密度（Density）
画面ごとに余白や行間を勝手に増減させず、プロジェクトで定義した密度から選択する。

- 密度スロット
    - `density-regular`
    - `density-compact`
- ルール
    - コンポーネント個別に `padding: 13px` のような独自調整は禁止
    - 密度によって変えてよいのは、余白/行高/ギャップなどの「密度に関わる値」のみ
    - 形状（radius など）や情報構造（並び順）は密度で変えない
    - `density-compact` は「説明文を省略しないと入らない」等の高密度表示が必要な場合にのみ使用する
        - 例: アイテムカードの説明文を表示しない（名称/主要スペックの可読性を優先）
        - スタック数/耐久度/主要プロパティは省略しない
#### 4.13.1 密度トークン
- `--density-space-mult`: `1.0` (regular) / `0.6` (compact)
- `--density-gap-mult`: `1.0` (regular) / `0.5` (compact)

### 4.14 ダイアログ規約（Modal / Dialog）
ダイアログは振る舞いと見た目を統一し、画面ごとに独自ルールを作らない。

- 用語
    - Modal: 背面操作をブロックするダイアログ
    - Sheet/Panel: 画面端から出るパネル（背面ブロックの有無は用途により定義）
- ルール
    - 開閉アニメーションはプロジェクトのモーションスロットから選ぶ
    - 閉じ方（×ボタン/外側クリック/ESC など）は種類ごとに固定し、画面ごとに変えない
    - オーバーレイ（背面暗幕）の濃さやぼかしはトークンで管理し、スタイル層で差し替える
    - ダイアログ内のスクロールは 4.9 のスクロールバー規約に従う
- スロット（例）
    - `dialog-modal`
    - `dialog-sheet`
    - `dialog-toast`（必要になったら追加）
#### 4.14.1 ダイアログ・トークン
- `--dialog-max-w`: `640px` (regular)
- `--dialog-max-h`: `80vh`
- `--overlay-opacity`: `0.8`
- `--overlay-blur`: `8px`

### 4.15 最大幅・最大高（Panel / Report / Receipt）
レシート/レポート/パネル系のUIは、画面ごとに伸縮ルールを変えず、最大幅・最大高を統一する。

- ルール
    - 最大幅・最大高はトークンで固定し、個別画面で独自値を作らない
    - 高さが溢れる場合は、コンテンツ領域をスクロールさせる（スクロールバー規約に従う）
    - 中央寄せ/余白（マージン）は共通ルールとし、スタイル差分は質感に限定する
- スロット（例）
    - `panel-narrow`（レシート/報告書）
    - `panel-regular`
    - `panel-wide`（必要になったら追加）

#### フッター情報 (.ui-footer)
- 用途: バージョン表記、コピーライトなどの補助情報。
- 形状: 画面左下に固定（absolute）。
- レイアウト: 垂直（column）に 2 行で表示。
- スタイル: 控えめな不透明度（0.4）と小さなフォントサイズ（0.75rem）で、メイン UI を邪魔しない質感。

#### 4.15.1 パネル・トークン
- `--panel-max-w-narrow`: `400px` (レシート/報告書)
- `--panel-max-w-regular`: `640px` (標準パネル)
- `--panel-max-w-wide`: `900px` (アーカイブ/大型パネル)
- `--panel-max-h`: `90vh`

### 4.16 ネオンパネル規約 (.ui-panel)
- **質感**: グラスモーフィズム（背景透過＋ぼかし）を必須とする。
- **背景色**: `rgba(255, 255, 255, 0.05)` 程度の微かな白み。
- **ボーダー**: サブカラー（ブルー）を `color-mix` で 60-70% 透過させたものを使用。
- **内部余白**: 枠自体は `padding: 0` とし、内部のコンテンツブロック側で余白を管理する。これにより、タブやセパレーターを枠いっぱいまで密着させることができる。

### 4.17 Launch ボタン規約 (.launch-button)
- **形状**: 特大のピル（radius-pill）形状。幅 340px 固定。
- **配色**: オレンジアクセント (`--color-accent`)。
- **質感**: 15%のアクセントカラー着色（`transparent 85%`）＋ グラスモーフィズム（`var(--blur)`）。背景画像の上でも高い視認性を担保する。
- **情報構造**: 
    - メインラベル (LAUNCH ENGINE)
    - サブテキスト (操作ヒント: ロケットと発射台を選択すると〜)
- **状態演出**:
    - **READY**: オレンジネオン点灯。
    - **OFF (Disabled)**: ボタン自体は消灯（グレーダウン）し、透過 95% の背景に減光するが、**サブテキストのみオレンジ色で明滅 (Blink)** させ、不備をプレイヤーに通知する。
    - **HOVER**: 内側に白い芯の輝きを追加し、外側に広大なブルームを放射。反転ではなく「眩しさの増幅」で表現。

### 4.19 パネル開閉ボタン規約 (.panel-toggle)
- **アクセシビリティ**: `button` タグは使用せず、`div` 等の非フォーカス要素で実装する。これにより、クリック時に意図しないフォーカスリング（枠線）が発生するのを防ぐ。
- **配置**: パネルのヘッダー（`.panel-tabs`）の右端に固定配置。
- **形状と質感**: 
    - 24px の円形（`radius-m`）。
    - 常に常時点灯の枠線（`--color-sub` の透過色）を持ち、背景透過とボカシを適用する。
- **開閉挙動 (Collapse)**:
    - パネルが閉じても開閉ボタンの表示位置は変えない。
    - 閉じた状態ではパネルのコンテントとタブを非表示にし、開閉ボタンそのものがフローティング・スイッチとして機能する。

### 4.18 アイテムカード詳細規約 (.item-card)
- **タイポグラフィ**:
    - 名称: `Outfit` / Bold / 14px / 白
    - メタ情報・プロパティ: `Courier Prime` / 10px / 透過白
- **選択状態 (`is-selected`)**:
    - カテゴリー色に基づく強い外周発光（Glow）と内部への滲み（Inner Shadow）を適用。
    - 枠線もカテゴリー色に変化する。
