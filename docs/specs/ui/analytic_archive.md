# UI Spec: Analytic Archive (記録画面) - V2

## 1. 目的と役割
プレイヤーのこれまでの航行記録、実績（Achievements）、およびリプレイデータを閲覧・分析するための統合ターミナル。宇宙船のメインシステムの一部として、高い情報密度と実用性を備えたデザインとする。

## 2. デザインコンセプト
- **スタイル**: 原則として「Matte（マット）」スタイルを適用。
- **レイアウト**: 最大 840px 幅（1120px から最適化）の 2 カラム構成（サイドバー + メインエリア）。
- **タイポグラフィ**: パネル全体の基準サイズを **14px** とし、各要素は `em` による相対指定でジャンプ率を管理（Title: 1.7em / Body: 1.0em / Caption: 0.7em）。
- **セマンティクス**: 
    - 統計情報には `.ui-stat-label` を使用。
    - 凡例には `.is-legend` 修飾子を付与し、カテゴリー色を反映したカラーラインを疑似要素（`::before`）で表示。

## 3. UI 構成要素

### 3.1 共通ヘッダー
- **タイトル**: 「Analytic Archive」およびサブタイトルを表示。`.ui-split-column` を使用。
- **タブ切り替え**: Analytics, Replays, Achievements の 3 つの主要カテゴリを切り替える `.ui-tab-group` を配置。
- **背景・境界線**: パネル本体と一体化させるため、背景は透明、境界線（セパレーター）は排除されている。

### 3.2 サイドバー (Left Pane)
- 累積統計（KPI）を一覧表示。
- **構造**: 各項目は `.ui-split-column` で構成され、ラベルを `.ui-well` の外に配置。
- **表示項目**: 累計突破セクター数、累計契約回数、総回収アイテム数、実績進捗率を表示する。最高到達セクターはサイドバーには表示せず、Analytics タブ内の Personal Best Ranking で扱う。
- **算出方法**: 累計突破セクター数、累計契約回数、総回収アイテム数は `GameRecordTracker` の `game_record_data` を参照する。実績進捗率は全実績定義に対する達成済み実績の割合とする。
- **初期表示**: ゲームプレイ結果が0件の場合、累計突破セクター数、累計契約回数、総回収アイテム数は `0`、実績進捗率は `0%` と表示する。
- **数値の質感**: `.num-sector` (Total Sectors Completed), `.num-contract` (Lifetime Contracts), `.num-item-count` (Total Items) などの意味論的クラスを使用し、標準的な数値フォントを適用。
- **サイズ**: 幅はコンテンツに応じた `auto` (PC時)。内部の `.ui-well` は **120px 固定幅** で中央寄せされる。
- **実績進捗 (Achievement Progress)**: サイドバー下部に、全実績に対する現在の達成率をパーセンテージ（例: 42%）で表示。

### 3.3 メインエリア (Right Pane / Main Content)
- **Analytics タブ**:
    - パフォーマンス推移グラフ: 直近20回のゲームプレイ結果を対象に、スコア・セクター・回収アイテム数の推移を視覚化。右上に `.is-legend` による凡例を表示。
    - 記録タイミング: ゲームプレイ結果は、契約終了が確定しゲームリザルト画面を表示するタイミングで1件追加される。
    - 保存項目: `RankTracker` は記録日時、最終スコア、踏破セクター数、到達セクター数、総回収アイテム数をランキング・直近表示用に保存する。累計KPIや実績判定用の通算記録は `GameRecordTracker` が保持する。
    - 対象外: 航行単位の成功/失敗/帰還/大破/遭難などの結果種別は Analytics では表示しない。
    - 平均線: 各系列について、直近20回のゲームプレイ結果の平均値を平均線として表示する。
    - 描画順: グラフ上では左を古い記録、右を新しい記録として表示する。保存配列の保持順と表示順は一致していなくてよい。
    - 履歴不足時: ゲームプレイ結果が20件未満の場合は、存在する件数のみを左から詰めて表示し、その件数で平均値を算出する。
    - 空状態: ゲームプレイ結果が0件の場合、グラフは空状態表示とする。Personal Best Ranking は空テーブルまたは `NO RECORDS` 表示とする。
    - ハイライト連動: Personal Best Ranking で選択中のカテゴリに対応する系列を、グラフ上でもハイライト表示する。
    - ベストランキングテーブル: 過去のゲームプレイ結果の最高記録をランキング形式で表示。カテゴリは `Score`, `Sector`, `Collected` の3つとし、各カテゴリ上位20件を表示対象とする。表示領域に収まらない分はスクロールで閲覧する。同値の場合は記録日時が新しいゲームプレイ結果を上位に表示する。
    - ゲームプレイ結果の保存配列: `Score` / `Sector` / `Collected` 各上位20件と、日時順の直近20件に含まれるゲームプレイ結果の和集合を保持する。同一記録が複数条件に含まれる場合は1件に統合する。
    - テーブル構造: `.ui-well` と `.is-scrollable` を組み合わせたスクロール可能なテーブル構造。
    - デザイン: 行間の境界線すら排除した、背景の明暗差のみによる極めてミニマルなフラットデザインを採用。
- **Replays タブ**:
    - 航行記録一覧: 最大20件のリプレイ記録を保持。お気に入り指定により削除保護が可能。
    - お気に入り枠: 最大5件まで登録可能。お気に入りはリプレイ保存枠20件の内数として扱う。
    - 自動保存枠: お気に入りが0件の場合は自動保存記録を最大20件まで保持できる。お気に入りが5件ある場合は、自動保存のみの記録を最大15件まで保持できる。
    - 自動保存判定: 保存数が20件未満の場合はスコアに関係なく自動保存する。保存数が20件に達している場合のみ、非お気に入り記録の最下位と新しい航行記録のスコアを比較して自動保存可否を判定する。同スコアの場合は新しい記録を優先する。
    - 保存枠の削除処理: 削除処理は21件目のリプレイ記録を保存しようとした場合のみ発生する。新しい航行記録が非お気に入り記録の最下位以上のスコアで自動保存対象になった場合、または最下位以下のスコアでも航行結果画面でお気に入り登録された場合に、保存後の非お気に入り最下位記録を削除する。
    - 表示件数と並び順: 保存されている最大20件をすべて表示する。並び順は獲得スコア降順とし、同スコアの場合は記録日時が新しい記録を上位に表示する。お気に入り状態は並び順に影響させず、星アイコン等の状態表示として扱う。
    - お気に入り上限時の操作: お気に入りが5件に達している状態で別の記録をお気に入り登録する場合は、航行結果画面と同じ置き換えダイアログを表示する。置き換え対象として選ばれた既存記録はお気に入り状態のみ解除され、即削除はしない。
    - お気に入り設定・解除: Replays タブでのお気に入り設定・解除では、新しいリプレイ記録の追加が発生しないため、保存枠20件を維持するための削除処理は行わない。お気に入り解除後の記録は、次回以降に新しいリプレイ記録が追加され、保存枠20件を超える場合の削除候補になる。
    - カラム構成:
        - **FAV**: お気に入り状態（★アイコン）。
        - **NO**: スコア順位。
        - **SECTOR**: 到達セクター。
        - **SCORE**: 最終スコア。
        - **DATE TIME**: 記録日時。
    - 一覧の情報量: 20件表示との相性を優先し、一覧はモック通りコンパクトな構成とする。発射構成（RocketItem / Launcher / Booster）はリプレイ再生中の構成パネルで確認できるようにする。
    - リプレイ再生中表示: 右側構成パネルには RocketItem / Launcher / Booster のみを表示する。セクター、スコア、記録日時は HUD 側に表示する。スコアは `1 tick = 1 point` として再生進行に同期して増加表示する。通常プレイで `LAUNCH` ボタンが配置される位置には、そのセクター内での帰還ボーナス（発射速度ボーナス）を表示する。
    - リプレイ再現方式: tick ごとの位置履歴は保存せず、保存された初期状態から物理計算を再実行して再生する。リプレイ記録には、同じ航行結果を再現するために必要な初期状態、発射構成、発射角度、セクター状態、帰還ボーナス（発射速度ボーナス）、物理再現に必要なデータを含める。発射構成はアイテムID参照ではなく、発射時点のスナップショットとして保存する。関係する各クラスは `getSnapshot()` で自身の再現に必要なデータをシリアライズし、factory の `fromSnapshot()` で復元される。親クラスの snapshot は所有する子オブジェクトの snapshot を内包し、復元時も子 snapshot から子インスタンスを復元する。保持物は `Item` 固定ではなく、実装上の `StackedItem` など該当クラスの snapshot 規約へ委譲する。snapshot 自体に型識別子や schema version は持たせず、復元時は所有構造に従って適切な `fromSnapshot()` へ渡す。将来的な snapshot 形式の非互換は、その snapshot を復元するクラスの `fromSnapshot()` 側で対応する。復元不能な snapshot は仕様・実装・データ整合性の異常として扱い、共通仕様の方針に従って例外として明示的に表出させる。具体フィールドは各クラス仕様で定義する。航行中に結果へ影響するランダム要素は存在しないため、リプレイ再現用の乱数 seed やランダム結果ログは保存しない。天体半径はランダム決定値のため保存し、天体質量は保存済み半径から再計算する。母星の半径・質量と出口アークの基本幅は固定マスタ値として扱い、リプレイ記録には変動値として保存しない。出口アークの再現には施設タイプと配置角度を保存し、幅拡大率は保存済み発射構成全体から再計算する。現時点では Booster が主な補正元だが、将来的に Booster 以外のアイテムが補正を持つ可能性があるため、特定カテゴリに固定しない。帰還ボーナスは `Sector` が保持するセクター状態の一部として保存する。
    - リプレイ中のマップ操作: リプレイ開始時のカメラ状態は、通常プレイ開始時と同じく、ユーザーが変更・永続化したカメラ状態を適用する。閲覧操作としてパン、ズーム、回転を許可する。リプレイ中のカメラ操作も通常プレイと同様に永続化し、次回以降の通常プレイおよびリプレイ開始時に適用する。これらは表示カメラにのみ影響し、リプレイの物理再生、スコア計算、航行結果には影響しない。構成変更、発射、AIM 操作などゲーム進行に影響する操作は無効化する。
    - 再生コントロール: 初期仕様では一時停止、再開、巻き戻し、シーク、速度変更などの再生コントロールは提供しない。再生速度は通常プレイと同じ `1x` 固定とする。リプレイは開始後、航行終了まで自動再生し、終了後は停止状態で `Replays` タブへ戻るボタンを表示する。
    - リプレイ終了時: 航行が終わった状態で停止し、最終セクターのマップ表示を維持する。ボタン操作で記録画面の `Replays` タブへ戻る。
    - TODO: 航行終了後に加算される各種ボーナスの表示方法を検討する。停止時に表示するのが望ましいが、中央のセクターマップを隠さない位置・サイズ・レイヤーで設計する必要がある。候補として、右側パネルまたは HUD 近辺の簡易ボーナスサマリー、中央を避けた小型オーバーレイがある。
    - TODO: 行選択時（`PLAY REPLAY` ボタンが有効になる状態）に、選択中リプレイの発射構成をプレビュー表示できるか検討する。モック画面上に表示領域があるか再確認してから最終決定する。
    - インタラクション:
        - 行を選択（クリック）することで、ヘッダー右側の「PLAY REPLAY」ボタンが有効化される。
        - お気に入りアイコンをクリックすることで、保護状態をトグル。
    - 空状態: リプレイ記録が0件の場合は、空テーブルまたは `NO REPLAYS` 表示とする。
- **Achievements タブ**:
    - 実績一覧を表示。各実績は「印刷スタイル（Overprinted Log）」カードとしてレンダリングされる。
    - **レイアウト**: `.achievement-showcase` 内で `display: grid` を使用。`minmax(300px, 1fr)` によるレスポンシブな多段カラム構成。
    - **視覚仕様**: 
        - 局所的な `.ui-style--printing` コンテキストを各カードに適用。
        - 達成済み実績には、右側に巨大な階級印（Tier Seal）が印字される。
        - ロック状態の実績はグレースケール（`grayscale(1)`）かつ半透明で表示され、タイトルは「NOT ACHIEVED」に差し替えられる。
    - **進行状況**: 各カード下部にインク密度の低い（透過）ゲージを表示し、次の Tier までの絶対進捗を視覚化。

## 4. レスポンシブ設計
- **ブレークポイント**: 800px 以下。
- **レイアウト変更**: サイドバーの `auto` 幅を解除し、1カラムの縦積み構成に変更。
- **KPIの最適化**: 4つの統計項目は、可能な限り横並び（`flex-row` / `wrap`）で表示され、省スペース化を図る。
- **スクロール制御**: 1カラム時はコンテンツが縦に長くなるため、`.ui-panel__body` に対して縦スクロールを有効化し、個別の `.is-scrollable` による制限を解除して全体で閲覧可能にする。この際のスクロールバースタイルは、プロジェクト標準のグローバル定義が自動適用される。

## 5. 実装構造
- **Base 層**: `ui_base.css`, `ui_layout.css`, `ui_primitives.css` を使用。
- **Style 層**: `ui_style_matte.css` による質感を適用。
- **Context 層**: `css/archive.css` で画面固有のレイアウト比率、中央寄せ、凡例装飾を定義。
