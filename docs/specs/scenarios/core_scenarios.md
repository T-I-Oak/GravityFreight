# Core Scenarios & Breakdown (コアシナリオとブレイクダウン)

## 概要
本ドキュメントは、Gravity Freight V2 の詳細設計を完了させるための「設計駆動装置」である。
各文脈を起点に UI、機能、シナリオをツリー形式でブレイクダウンし、実装に必要なクラスインターフェースを導き出す。
検討済みの項目にはチェックを入れ、進捗管理に使用する。

### ノード分類
- **[UI]**: 表示項目、レイアウト要素、装飾。
- **[機能]**: 内部ロジック、データ処理、計算。
- **[シナリオ]**: ユーザー体験、状態遷移、イベント連鎖。

## 8 ゲームオーバー画面
**役割**: ゲーム終了（契約終了）時の退場演出、終了理由の表示。
- [x] 8.1 [機能] 終了理由の表示

- [x] 8.2 [機能] 退場演出（逆方向ワープ等）
    - **遷移元**: ゲームオーバー画面は、航行結果画面（第5章）および施設画面（第6章）から遷移してくる。
    - **判定責務**:
        - ゲームオーバー判定ロジックは `EconomySystem.checkGameOver(sessionState)` に集約する。
        - 航行結果確定後と施設退出時の両方で、同じ判定ロジックを利用する。
    - **通常遷移との差分**:
        - 通常は航行結果画面または施設画面の処理完了後、母星に戻って航行準備（第3章）へ入る。
        - ゲームオーバー条件を満たしている場合は、航行準備を開始せず、現在のマップから遠ざかる退場演出へ移行する。
    - **退場演出**:
        - セクター開始シーンのワープ演出と対になる、逆方向ワープとして表現する。
        - 通常のセクター開始ワープは、直前セクターのマップが表示された状態からズームインして次セクター方向へ進む演出である。
        - ゲームオーバー時はその逆として、直前セクターのマップが表示された状態からズームアウトし、スタートのセクター方向へ戻っていくように見せる。
        - カメラまたはマップ表示は、母星周辺の航行準備位置へ近づくのではなく、現在のマップ全体から離脱する方向へ移動・縮退する。
        - 演出序盤は背景・マップの表示を維持し、プレイヤーが「通常遷移とは逆方向に進んでいる」ことを認識できるようにする。
        - マップは一定サイズまで縮退した後、中央に残り続けないように薄くして消す。
    - **ゲームリザルトの重ね表示**:
        - 逆方向ワープ中に、第7章のゲームリザルト画面をレシート形式のオーバーレイ（パネル）として表示する。
        - レシート表示はワープ開始と同時ではなく、逆方向ワープが視認できる短い遅延後に開始する。
        - レシート表示時、外側を暗い印刷オーバーレイで覆わず、逆ワープ中の星背景の上に紙面だけを重ねる。
        - レシート本文には `SECTORS COMPLETED`、`TOTAL COLLECTED`、`FINAL SCORE` と、それぞれの ranking / grade を表示する。ゲームオーバー理由は score の ranking / grade の代替表示にしない。
        - ゲームリザルトには累計スコア、総獲得コイン、踏破セクター数、合計航行時間など、7.1で定義した最終結果を表示する。
        - 退場演出はリザルト表示の背景演出として扱い、リザルトパネルの操作を妨げない。
    - **タイトル復帰**:
        - ゲームリザルト画面には、モックと同様にタイトルへ戻るボタンを表示する。
        - ボタン文言は `END CONTRACT` とする。
        - ボタン押下時、逆方向ワープを減速させて通常速度へ戻してから、`AppOrchestrator.returnToTitle()` を呼び出し、現在のゲームライフサイクルを終了してタイトル画面（第1章）へ戻る。
        - タイトルへ戻る際、現在の `SessionState` は破棄される。ただし、記録・実績はゲームリザルト表示時点で、リプレイは航行終了時点で確定済みであること。

            - 行選択時に、選択中リプレイの発射構成プレビューは表示しない。発射構成はリプレイ再生中の構成パネルで確認する。
            - リプレイ記録が0件の場合は、空テーブルまたは `NO REPLAYS` 表示とする。
- [ ] 10.2 [機能] リプレイ再生の管理
    - **保存単位**:
        - リプレイは1回の航行単位で保存する。
        - リプレイは tick ごとの位置履歴を保存せず、保存された初期状態から物理計算を再実行して再生する。
        - 保存対象には、その航行の初期状態、発射構成、発射角度、セクター状態、帰還ボーナス（発射速度ボーナス）、物理再現に必要なデータ、航行結果を含める。
        - 物理再現に必要なデータは、同じ初期状態から同じ航行結果を再現できることを要件とする。
        - 航行中に結果へ影響するランダム要素は存在しないため、リプレイ再現用の乱数 seed やランダム結果ログは保存しない。
        - 発射構成はアイテムID参照だけでは不十分なため、発射時点のスナップショットとして保存する。
        - リプレイ保存データは、関係する各クラスの `getSnapshot()` によって生成されたスナップショットを集約して構成する。
        - リプレイ再生時は、各クラスの factory である `fromSnapshot()` にスナップショットを渡してインスタンスを復元する。
        - シナリオ仕様では「リプレイ再現に必要な状態をスナップショット化する」ことを要件とし、各クラスを再現するために必要な具体フィールドは、それぞれのクラス仕様で定義する。
        - 親クラスの `getSnapshot()` は、自身が所有する子オブジェクトの `getSnapshot()` 結果を内包する。
        - `fromSnapshot()` は、内包された子オブジェクトの snapshot から子インスタンスも復元する。
        - 例: `Sector` は所有する `CelestialBody` / `ExitArc` の snapshot を含み、`CelestialBody` は保持物の snapshot を含む。
        - 保持物は必ずしも `Item` 固定ではなく、実装上は `StackedItem` などの別クラスである可能性がある。親クラスは保持している実クラスの snapshot 生成・復元規約に委譲する。
        - snapshot 自体に `snapshotType` や `className` のような型識別子は持たせない。
        - snapshot を生成したクラス自身がその形式を知っているため、復元時は所有構造に従って適切な `fromSnapshot()` へ渡す。
        - snapshot 自体に schema version などのバージョン情報は持たせない。
        - 将来的に snapshot 形式の非互換が発生した場合は、その snapshot を復元するクラスの `fromSnapshot()` 側で対応する。
        - `fromSnapshot()` で復元できない snapshot は、環境的な問題ではなく仕様・実装・データ整合性の異常として扱う。
        - 共通仕様の方針に従い、この異常は再生不可表示などで包み隠さず、例外として明示的に表出させる。
        - ネットワーク等の環境要因による失敗は別扱いとし、ユーザー体験を保つためのエラーハンドリング対象とする。
        - 帰還ボーナス（発射速度ボーナス）は `Sector` が保持するセクター状態の一部として保存する。
        - セクター内の天体半径はランダムに決定されるため、リプレイ記録に保存する。
        - 天体質量は半径から決定されるため、保存済み半径から再計算できる値として扱う。
        - 母星の半径・質量は固定マスタ値として扱い、リプレイ記録には変動値として保存しない。
        - 出口アークの基本幅は施設タイプで固定されるため、リプレイ記録には変動値として保存しない。
        - 出口アークの再現に必要なのは配置角度である。
        - 出口アーク幅の拡大率は発射時の構成全体から決まる補正値として扱い、保存済み発射構成から再計算する。
        - 現時点では Booster が主な補正元だが、将来的に Booster 以外のアイテムが出口幅拡大率補正を持つ可能性があるため、特定カテゴリに固定しない。
        - セクター状態は `Sector.getSnapshot()` で保存し、`Sector.fromSnapshot()` で復元する。スナップショットに含める具体フィールドは `Sector` のクラス仕様で定義する。
    - **自動保存ルール**:
        - 1回の航行で取得したスコアを基準に、スコア上位の航行記録を自動保存する。
        - リプレイ保存枠の総数は最大20件とする。
        - お気に入り登録数は最大5件とし、リプレイ保存枠20件の内数として扱う。
        - お気に入りが0件の場合、自動保存記録は最大20件まで保持できる。お気に入りが5件ある場合、自動保存のみの記録は最大15件まで保持できる。
        - リプレイ保存数が20件未満の場合、新しい航行記録はスコアに関係なく自動保存する。
        - リプレイ保存数が20件に達している場合のみ、非お気に入り記録の最下位と新しい航行記録のスコアを比較して自動保存可否を判定する。
        - 新しい航行記録が非お気に入り記録の最下位と同スコアの場合は、新しい記録を優先して自動保存対象とする。
        - 21件目のリプレイ記録を保存しようとした場合のみ、保存枠20件を維持するための削除処理を行う。
        - 21件目の保存が発生するのは、新しい航行記録を追加する場面である。
        - 新しい航行記録が、非お気に入り記録の最下位以上のスコアで自動保存対象になった場合、保存後に非お気に入り記録の最下位を削除する。
        - 新しい航行記録が、非お気に入り記録の最下位以下のスコアであっても、航行結果画面でお気に入り登録された場合は保存対象になる。この場合も保存後に非お気に入り記録の最下位を削除する。
        - 非お気に入り記録の最下位判定はスコア昇順、同スコアの場合は記録日時が古い順とする。
    - **お気に入り登録**:
        - 航行結果画面で、対象の航行記録をお気に入りに設定できる。
        - リプレイ画面でも、お気に入りの設定・解除ができる。
        - お気に入り登録された記録は、自動保存枠の置き換え対象から除外する。
        - お気に入り登録の上限数は最大5件とする。
        - 航行結果画面では、対象の航行記録が自動保存対象外であっても、お気に入り登録によって保存対象にできる。
        - 新しい航行記録が自動保存対象外であり、航行結果画面でお気に入り登録もされなかった場合、そのリプレイデータは保存しない。航行結果画面を抜けた時点で破棄する。
        - お気に入り上限5件に達している状態で新しい記録をお気に入り登録しようとした場合、既存のお気に入りから置き換え対象を選ぶダイアログを表示する。
        - リプレイ画面からのお気に入り登録でも、同じ置き換えダイアログを呼び出す。
        - 置き換えが確定した場合、選択された既存記録はお気に入り状態を解除され、新しい記録がお気に入りとして保存される。
        - お気に入りを外された既存記録は、その時点で即削除しない。
        - 新しいリプレイ記録の追加によりリプレイ保存枠20件を超える場合、非お気に入り記録の中からスコア最下位の記録を削除する。同スコアの場合は記録日時が古い記録を削除する。
        - そのため、置き換えダイアログで選択された記録が削除されるとは限らない。
        - 既存のお気に入りを単に解除した場合、その記録は即削除せず、通常の非お気に入り記録として保持する。
        - リプレイ画面でのお気に入り設定・解除では、新しいリプレイ記録の追加が発生しないため、保存枠20件を維持するための削除処理は行わない。
        - お気に入り解除後の記録は、次回以降に新しいリプレイ記録が追加され、保存枠20件を超える場合の削除候補になる。
    - **リプレイ一覧の表示内容**:
        - 獲得スコア
        - 到達セクター数
        - 記録日付
        - お気に入り状態
        - 一覧は最大20件すべてを表示し、獲得スコア降順、同スコアの場合は記録日時が新しい順に並べる。
        - お気に入り状態は並び順に影響させない。
        - 一覧はモック通りコンパクトな構成を基本とし、発射構成（RocketItem / Launcher / Booster）は再生中の構成表示パネルで確認できるようにする。
        - リプレイ行選択時に、選択中記録の発射構成プレビューは表示しない。
    - **再生画面**:
        - リプレイ再生は全画面で行う。
        - 通常プレイと同じマップ表示を使い、航行軌道、重力干渉、到達/失敗の流れを通常プレイと同じ空間スケールで確認できるようにする。
        - 右側のビルドパネル領域には、発射構成（RocketItem / Launcher / Booster）を表示する。
        - セクター、スコア（リアルタイム表示）、記録日時は HUD 側に表示する。
        - リプレイ中のスコアは、航行時と同じく `1 tick = 1 point` として再生進行に同期して増加表示する。
        - 通常プレイで `LAUNCH` ボタンが配置される位置には、そのセクター内での帰還ボーナス（発射速度ボーナス）を表示する。
    - **再生中の構成表示**:
        - 通常プレイ画面の右側ビルドパネル相当の領域を、リプレイ中は構成表示パネルとして使用する。
        - 構成表示パネルには、使用した RocketItem、Launcher、Booster を通常のアイテムカード表現で表示する。
        - リプレイ中は構成の編集や発射操作はできない。
    - **リプレイ中のマップ操作**:
        - リプレイ開始時のカメラ状態は、通常プレイ開始時と同じく、ユーザーが変更・永続化したカメラ状態を適用する。
        - リプレイ再生中も、閲覧操作としてマップのパン、ズーム、回転を許可する。
        - リプレイ中のパン、ズーム、回転操作も通常プレイと同様に永続化し、次回以降の通常プレイおよびリプレイ開始時に適用する。
        - これらの操作は表示カメラにのみ影響し、リプレイの物理再生、スコア計算、航行結果には影響しない。
        - 構成変更、発射、AIM 操作など、ゲーム進行に影響する操作は無効化する。
    - **再生コントロール**:
        - 初期仕様では、一時停止、再開、巻き戻し、シークなどの再生コントロールは提供しない。
        - 再生速度は通常プレイと同じ `1x` 固定とし、速度変更操作は提供しない。
        - リプレイは開始後、航行終了まで自動で再生する。
        - 航行終了後は停止状態となり、記録画面の `Replays` タブへ戻るボタンを表示する。
    - **リプレイ終了時のボーナス表示**:
        - 航行終了後に加算される各種ボーナスは、通常プレイでは航行結果画面で表示される。
        - リプレイは航行が終わった状態で停止し、ボタン操作で記録画面の `Replays` タブへ戻る。
        - 停止状態では最終セクターのマップ表示を維持する。
        - リプレイ終了時に、航行終了後ボーナスの追加表示は行わない。
