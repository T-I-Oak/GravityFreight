# Gravity Freight V2: Core Development Philosophy

V2開発におけるあらゆる意思決定（設計、コーディング、ドキュメンテーション）は、以下の三原則を絶対的な基準とする。

---

## 1. 共通化 (Standardization / Commonality)
「一つしかないものは、一つしか書かない」
- 重複は悪であり、技術的負債の源泉である。
- ロジック、スタイル、定数は可能な限り抽象化し、再利用可能な形で中心（Core）に集約する。
- 複数の場所で同じ情報が必要な場合、必ず「Single Source of Truth（唯一の真実）」を定義する。

## 2. 階層化 (Layering)
「役割を混ぜない、境界を越えない」
- システムは明確な階層（レイヤー）で構成される。
- 上位レイヤー（具体的・表情的）は、下位レイヤー（抽象的・基礎的）の定義を破壊してはならない。
- UI Architecture:
    - Tokens (意味・定数) -> Base (能力・機能) -> Style (装飾・表情)
- Code Architecture:
    - State (データ・真実) -> System (法・処理) -> View (表現・演出)

## 3. 意味論 (Semantics)
「手順（How）ではなく、意図（What）を記述する」

### 3.1 命名による定義 (Naming as Meaning)
- 定数: 4000 という生の値を使わず、const GRAVITY_G = 4000 のように、その数値が何を表すのかを命名によって定義する。
- 変数・関数: 実装手法（Array, Map）ではなく、情報の種類（SectorHistory, CargoList）で命名する。

### 3.2 抽象化による意図の宣言 (Declarative Logic)
- 直接的な低層演算（score += 10）を避け、意味のある振る舞い（addScore(10)）として抽象化する。
- 利点:
    - コードが物語（意図）として読めるようになる。
    - 演算そのものに加え、関連する副次処理（UI更新、実績判定、リダンダンシーの確保など）を一つの「意味ある動作」として隠蔽・管理できる。

## 4. ノイズの排除と純度の維持 (Noise Elimination)
ドキュメントおよびソースコードは、常に「最新の真実」のみをノイズなく表現しなければならない。

### 4.1 過剰な装飾の禁止
- 強調（太字）の抑制: 太字は多用せず、それがなければ意味が伝わらない重要な箇所にのみ限定する。
- 絵文字の禁止: 視覚的なノイズとなり情報の質感を損なうため、絵文字の使用を全面的に禁止する。

### 4.2 履歴・比較の排除
- ドキュメントに「以前より〜した」といった過去との比較を記載しない。
- 過去の経緯はGit等の履歴が管理するものであり、ドキュメントは「現状がなぜこうあるべきか（Why）」という論理的な根拠に集中する。

### 4.3 コメントにおける「Why」の記述
- ソースコードに実装手順（How）をコメントとして記載することは、コードの重複であり無意味である。
- コメントは、コードから読み取ることが難しい「背景」「意図」「選択の理由（Why）」に限定し、純度を高く保つ。
