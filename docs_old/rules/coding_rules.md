# コーディング規約 (Coding Rules)

本ドキュメントは、プロジェクト全体のコード品質とデザインの一貫性を保つためのルールを定めます。

## 1. CSS (Styling)

### 1.1 フォルダの構成と責務 (CSS Structure)
スタイルの肥大化と衝突を防ぐため、以下の役割分担に従ってファイルを分割・管理すること。

- **`base.css`**: 全体の CSS 変数（`:root`）、基本的なリセット定義。
- **`animations.css`**: 全画面共通で使用されるキーフレームアニメーションや視覚効果の定義。
- **`layout.css`**: 画面全体のレイアウト構造に関する定義。
- **`ui.css`**: 複数の画面で再利用される「極めて汎用的な」UI パーツ。
- **`buttons.css`**: ゲーム内の共通ボタンのデザイン定義。
- **`item-card.css`**: インベントリや報酬画面等で全画面共通で使われる「アイテムカード」専用。
- **`[screen/system-name].css`**:
    - 特定の画面やシステムに閉じたスタイル（例: `mission-hud.css`, `result-screen.css`, `story-overlay.css`）。
    - **原則**: 新しい画面や大規模な機能を追加する場合、`ui.css` に追記せず、専用のファイルを新規作成してカプセル化すること。

### 1.2 原則
- **!important の禁止**: メンテナンス性確保のため、原則として `!important` を使用しない。
- **インラインスタイルの禁止**: HTML 内の `style="..."` は動的なアニメーションなどの特別な理由がない限り原則的に禁止。
- **スコープの継承 (Scoped CSS)**:
    - HTML 要素のスタイリングは、可能な限り上位ノードの ID を使用してカプセル化（隔離）すること。
    - 理由: 予期せぬスタイルの波及（Side-effect）を完全に防ぐため。
- **状態表示の命名標準化 (Status Badge Naming)**:
    - 報酬カードや HUD の状態表示クラスなどは、可能な限り `status-[state]`（例: `status-delivered`, `status-new`）で統一すること。

## 2. JavaScript (Logic)

### 2.1 設計・実装原則
- **「念のため」のガードの禁止**: 実装の不備を隠蔽する防御的コード（プロパティ存在チェックの乱用など）を禁止。不正な値には例外を投げる、シミュレーション停止などを行い不具合を即座に顕在化させること。
- **データの正規化と一元化 (Data Centralization)**:
    - 施設の色（`GOAL_COLORS`）、名称、カテゴリカラー、共通定数などは、必ず **`Data.js`** を参照すること。
    - 各ロジック内での色コードや名称文字列の**ハードコードは厳禁**とする。
- **デバッグコードの完全削除 (No Debug Code)**:
    - 検証用の `console.log` やデバッグ表示用ビジュアライザは、マージ・コミット前に必ずすべて削除、または無効化すること。
- **インタラクティブ要素の範囲制限 (Interactive Target Scoping)**:
    - クリックイベント等は、親要素全体などの曖昧な範囲ではなく、特定のサブ要素（アイコン等）に限定して設定すること。
- **ソースコード内コメントの原則 (Commenting Rules)**:
    - コメントには、コード自体からは読み取れない**「実装の意図（Why）」**や、複雑なアルゴリズムの解説のみを記載すること。
    - **原則**: 「指摘されたので修正」「前回の 1.5 倍にする」といった、開発の経緯や以前の状態との比較などの**相対的な情報は「ノイズ」となるため記述を禁止**する。
    - コードは常に「現在の仕様」を正しく表現するべきであり、調整が必要な場合は「W=30 // 視認性確保のための最適値」のように、絶対的な意図を記載すること。

### 2.2 グローバル管理
- グローバル変数（`window.game` 以外の汚染）を禁止。常に上位クラス、またはモジュールスコープで管理すること。
