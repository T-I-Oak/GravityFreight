# Gravity Freight 開発プロセス運用ルール

本書は、Gravity Freight の開発を進める際の「コミュニケーション方針」「進行手順」「完了判定」をまとめた運用ルールです。

## 基本ルール
- チャットおよび `Implementation Plan` や `Task` など、すべてのユーザーインタラクションは日本語を基本とする。

## 開発ルール
- 開発開始時
  - 開発要件に即したブランチを作成し、開発はブランチ上で作業すること
  - `tmp_` もしくは `local_` で始まるファイル・フォルダーは `.gitignore` で管理対象外にすること

- 開発中
  - [重要!!] Windows環境であるためコマンドを実行するときは `&&` などを用いないこと
    - これは `Implementation Plan` に記載していつでも思い出せるようにしておくこと
  - 一時ファイルを作成する場合はファイルもしくはディレクトリ名の先頭に `tmp_` を付けた名称にして、開発が終わるまでに削除すること
  - 一時ファイルではないがGitにpushしないファイルもしくはディレクトリ名はファイルの先頭に `local_` を付けた名称にすること
  - 1ファイル当たり 500step 程度以内を心がけるようにして、それを超えた場合はリファクタリングを提案すること
  - `docs` フォルダーに仕様を作成し、適宜最新化するとともに実装が仕様から乖離していないことをチェックすること
  - `tests` フォルダーにテストスクリプトを作成し、適宜テストがパスすることを確認すること
  - テストフレームワークは開発言語のデファクトスタンダードなフレームワークを使用すること

- 開発完了時
  - 今回の開発で `docs` に記載すべき仕様があれば記載する
  - Documentが最新化されていることを確認する
  - 今回の開発で追加すべきテストがあれば追加する
  - テストが最新化されていることを確認する
  - すべてのテストがパスすることを確認する
  - 今回完了したバックログを特定し、リストをユーザーに通知する
  - [重要!!] 開発完了時はユーザーに承認を得ること
    - これは `Implementation Plan` に記載していつでも思い出せるようにしておくこと
    - `Task` には `[STOP!] 開発完了時にユーザーの承認を得る` などを記載し、確実に止まるようにする
