## 0. 全ドキュメント共通の原則 (Global Standards)

1. **ノイズの徹底排除**:
   - 仕様書やアーカイブは事実と思い出を記録する場所ではなく、「必要な情報」を素早く提供する場です。ソースコードを読めば分かる自明なパラメーター構成や、単なる作業の経緯（「〇〇をリファクタリングした」「テストを追加した」等）はノイズとなるため記載しません。
   - 開発者の自己アピール（作業ログ）ではなく、次に見る人のための客観的な状態を記述してください。
2. **装飾・強調（Bold）の制限**:
   - `**太字**` や絵文字（🚀等）による過剰な装飾は原則禁止します。
   - みんなが太字を使い始めると、ドキュメントの本質が視覚的ノイズに埋もれます。強調は「本当に致命的な警告」やディレクトリ構造の明示など、構造上不可欠な場所にのみ使用してください。

## 1. 仕様書 (`docs/specs/`) の役割分担と構成

### 1.1 フォルダ構成ルール
雑多なドキュメントの蓄積（負の連鎖）を防ぐため、以下の構成を厳守します。**ルート直下にはファイルを置いてはなりません。**

- **`docs/specs/requirements/`**: 
    - ゲームの不変のゴール、ルール、体験を定義する「要求仕様」を集約する最重要フォルダ。
- **`docs/specs/systems/`**: 
    - 特定のゲームシステム（ストーリー、経済、ミッション等）の機能詳細仕様。
- **`docs/specs/ui/`**: 
    - 各画面のレイアウト、演出、操作ガイド等のUI/UX仕様。
- **`docs/specs/technical/`**: 
    - 実装上の制約、アーキテクチャ、技術的な掟を定義。

### 1.2 要求仕様書 (`docs/specs/requirements/`)
**【役割】**
実装（ソースコードやプラットフォーム）に依存しない、純粋な「ゲームとしてのルールと体験（What）」を厳格に定義するマスタードキュメント群。
要求（「あるべき姿」）を不変のものとして定義し、プロジェクト・リード（ユーザー）の承認なしに内容を書き換えることはできません。

**【構成ファイル】**
- **`core_mechanics.md`**: 基本サイクル、物理法則の計算式、勝利・失敗条件。
- **`item_catalog.md`**: アイテムのカテゴリー定義、具体的な性能数値（スペック）と ID の一覧。
- **`world_config.md`**: 初期設定、出口（報酬）、ドロップ率、施設機能などのシステム定数。

**【記載しないもの】**
- 詳細な CSS クラス名、内部的なメソッド名、一時的なデバッグ用フラグなどの実装依存情報。

---

### 1.3 `technical/` 配下 (技術および実装のメタ仕様書)
**【役割】**
「次に実装する人がバグを作り込んだり、デザインの一貫性を壊したりしないため」に守るべき、設計上の制約やメタルール（Why/How）を定義する。

**【記載するもの】**
- UI の責任境界（例：「.hidden の管理責任はこのスクリプトにある」などのアーキテクチャ上の掟）。
- デザインの運用メタ情報（例：具体的なカラーコード一覧ではなく、「アイテムのカテゴリカラーはUIの縁取りやパーティクルにのみ使用し、視認性のためテキスト自体には適用しない」といった適用ルール）。

**【記載しないもの】**
- 「ソースコードを見れば自明なこと」（カラーコードの羅列、配列の中身、単純な処理の流れのトレース）。

---

### 1.4 メモおよびレポートの扱い
- 開発中の調査結果や一時的なレポート（例：`ui_terminal_report.md` のような設計メモ）は、**「最終的に `ui/` や `systems/` 下の正式な仕様書へ統合」**するか、不要になった時点で削除すること。
- 仕様書フォルダーを「作業経過のゴミ箱」にしてはならない。

## 2. 計画・履歴ドキュメント (`docs/planning/`)

### 2.1 `backlog.md` (開発バックログ)
**【役割】**
今後着手すべき未了タスクのプール。

### 2.2 `backlog_archive.md` (改版履歴 / Changelog)
**【役割】**
「そのバージョンでユーザーの体験がどう変わったか」のみを記述する記録簿。

**【記載基準と制限】**
- **ユーザー体験の変更のみ記載**: 機能追加、ゲームバランスの変更、UIの改善、ユーザー面に影響するバグの修正のみを記録する。
- **ノイズの排除**: コードのリファクタリング、テストの追加、変数名の変更など「内部的な品質向上や作業」は箇条書きのリストや本文に含めない。ただし、内部作業のみで完了したバージョンにおいてバージョン番号の飛びを防ぐため、バージョンの見出し（タイトル）のみを作成することは例外として必須とする（本文は不要）。
- **フォーマット**: 絵文字は使用せず、太字などの装飾も排除したプレーンな箇条書き（`- 項目名: 変更の要約`）とする。

```markdown
## Version [Major].[Minor] (YYYY/MM/DD) - [バージョンの主目標]

### 機能追加 (Feature)
- [項目名]: 何ができるようになったかを簡潔に記載。

### 機能改善 (Enhancement)
- [項目名]: UIやUX、バランスがどう良くなったかを記載。

### 不具合修正 (Bugfix)
- [項目名]: 何のバグがどう直ったかを記載。
```
