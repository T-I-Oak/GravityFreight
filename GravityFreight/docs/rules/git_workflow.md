# Git ワークフロー (Git Workflow)

本ドキュメントは、本プロジェクトにおけるブランチ管理、バージョン、リリース手順について定めます。

## 1. バージョン管理

### 1.1 バージョン表記
セマンティックバージョニング（Major.Minor.Patch）に準拠して管理します。
- `GravityFreight/src/core/Game.js` の `VERSION` 定数、`package.json` の `version` フィールド、および `index.html` に同一のバージョンを記述すること。

### 1.2 バージョン更新ルール
- **Major (メジャー)**: ユーザーから明示的な指示があった場合のみ。
- **Minor (マイナー)**: **「新しい開発（新ブランチ）」を開始する際**に現在のバージョンに1加える。
    - 例: 現在 `0.16.x` であれば、次の開発ブランチは `v0.17` となる。
- **Patch (パッチ)**: JSファイル等のコードを修正した場合、ブラウザやテストランナーのキャッシュによる悪影響を防ぐため、**「テストを実行する前」**に必ずパッチバージョンを1つ上げること。

### 1.3 キャッシュ無効化 (Cache Invalidation)
`index.html` での JS 読み込み時にクエリパラメータを付与します。
- `index.html` 内の `<script>` タグの末尾に `?v=X.X.X` の形式でバージョンを指定すること。

## 2. ブランチ命名規則 (Branch Naming)
以下のプレフィックスを基本とし、開発対象のバージョン番号（**`v[Major].[Minor]`** 形式）を含めてください。
- **重要**: ブランチ名は「現在のマイナーバージョンに1加えたもの」を原則とします。
- **注意**: 開発中にパッチバージョンが変動するため、ブランチ名にはパッチバージョン（末尾の数値）を含めないこと。
- **新規機能**: `feature/v[Major].[Minor]-[description]`（例: `feature/v0.17-story-system`）
- **リフレッシュ**: `refactor/v[Major].[Minor]-[description]`（例: `refactor/v0.17-maintenance`）
- **バグ修正**: `fix/v[Major].[Minor]-[description]`（例: `fix/v0.17-quick-fixes`）

## 3. マージ & プッシュ手順 (Final Checklist)
マージを行う前に、以下の項目をすべて完了させてください。

1. **ブランチの作成**: バージョン（Major.Minor）を冠したブランチを作成。
2. **小規模なコミット**: 意味のある単位で、正確な説明を付けてコミット。
3. **テスト実行とバージョン同期**: 
    - テストにおけるキャッシュ問題を回避するため、**テスト実行前にパッチバージョンを上げ**、`Game.js`, `index.html`, `package.json` の3箇所でバージョン番号を同期させること。
    - 新機能や変更点に対して必要なテストケースが追加されていること。
    - `npm test` を実行し、既存テストを含む全件が合格（グリーン）していること。
5. **要求仕様書の同期**: 今回の開発内容が `docs/specs/requirements/` 内の各ドキュメント等のユーザー要求に背いていないか確認し、実装と仕様の整合性を保つこと。
6. **バックログの移行**: 完了した項目を backlog.md からアーカイブへ移動すること。

## 4. バックログアーカイブの運用 (Archiving Rule)

完了したタスクのアーカイブ方法、および `backlog_archive.md` の記載フォーマットについては、新たに定義された `docs/rules/document_standards.md` の「2.2 backlog_archive.md (改版履歴 / Changelog)」セクションの準拠ルールに従ってください。

## 7. テスト構成 (Testing Policy)

テストは役割に応じて `tests/` フォルダ下に階層化してください。
- **`tests/spec/` (仕様テスト)**: 仕様書（`docs/specs/`）に完全に準拠した連番付きの受入テスト（例: `01_xxx.test.js`）。
- **`tests/implementation/` (実装テスト)**: 各モジュールの内部ロジックやデータ整合性を検証するユニットテスト。
- **マージ条件**: `npm test` を実行し、全テストが合格（グリーン）している結果を提示すること。
