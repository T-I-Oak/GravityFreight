# テストポリシ (Testing Policy)

本ドキュメントは、本プロジェクトにおけるテストの構成、ツール、実行基準について定めます。

## 1. テストフォルダの構成 (Folder Structure)

テストは役割に応じて `tests/` フォルダ下に階層化し、整理すること。

- **`tests/spec/` (仕様・受入テスト)**:
    - **命名規則**: `spec.md` 内の章構成（例: `01_xxx.test.js`, `02_yyy.test.js`）と**完全に対応させた名称**を付けること。
    - **内容**: 仕様書で定義された挙動を満たしているかを確認する受入テスト。
- **`tests/implementation/` (ユニット・実装テスト)**:
    - **命名規則**: `src/` フォルダ内のディレクトリ構成およびファイル名と **1 対 1 で対応させた名称**を付けること。
    - **例**: `src/core/Game.js` のテストは `tests/implementation/core/Game.test.js` に配置。
    - **内容**: 各モジュールの内部ロジック、データ整合性を検証。
- **`tests/test-utils.js` (テストユーティリティ)**:
    - モックデータ、ヘルパー関数等、テスト間で共通で利用するコードを管理。

## 2. フレームワークと実行ツール

- **テストツール**: **`vitest`** を使用。
- **実行基準**:
    - 全テストの実行: `npm test`
    - 特定ファイルの実行: `npx vitest [path/to/test]`
    - **マージ前**: `npm test` を全パスさせてグリーン（合格）を確認すること。

## 3. テストの作成指針 (Testing Strategy)

### 3.1 TDD (テスト駆動開発) の原則適用
複雑な計算ロジック、状態管理、多システム間のイベント連鎖などが含まれる実装において、**「テストコード（仕様定義）を先に書き、実装を後にする（TDD）」** アプローチを原則義務とする。
- **例外**: 単純なテキスト修正、色の変更、および自明な UI の微調整を除き、本工程を標準のプロセスとする。
- **目的**: 境界値（Edge cases）に対する堅牢な設計を強制し、デバッグ工数の削減と品質保証を同時に達成するため。

### 3.2 設計の標準化
- **AAA パターン**: 一般的なテストのベストプラクティス（Arrange-Act-Assert パターン等）に従うこと。
- **副作用のテスト**: `EventSystem` や `StorySystem` 等の他システムとの連鎖反応を確認すること。
- **境界値の徹底検証**: 異常値、最小・最大値を重点的にテストすること。
