# 技術仕様書 (Technical Specification) - UI & Implementation

本ドキュメントは、Gravity Freight の UI 実装における技術的な制約、レイヤー構造、および設計意図をまとめたものです。ソースコードを読んだだけでは判別しにくい重要な数値を明文化し、将来のデグレーションを防ぐことを目的とします。

## 1. UI レイヤー構造 (Z-index 定義)

UI の重なり順は `z-index` によって厳密に管理されています。新しいオーバーレイやボタンを追加する際は、以下の階層を守る必要があります。

| Z-index | 対象要素 / カテゴリ | 理由 |
| :--- | :--- | :--- |
| **7000** | **最前面アクションボタン** | `launch-control`, `.result-actions`。いかなる演出（レシート等）よりも前面に配置し、常に操作可能にするため。 |
| **6000** | **演出用オーバーレイ** | `receipt-overlay` (レシート)。リザルト画面の上に重なる一時的な演出用。 |
| **5000** | **メインオーバーレイ** | `result-overlay`, `event-overlay` (Title Screen も含む)。ゲームプレイを遮断する主要な UI。 |
| **1000** | **UI コントロールコンテナ** | `#launch-control` の親など。HUD よりも前面だが、オーバーレイよりは背面。 |
| **50 - 100** | **恒常 HUD / パネル** | `mission-hud`, `build-overlay`。プレイ中常に表示される情報レイヤー。 |
| **0 (Default)** | **ゲームキャンバス** | `bg-canvas`, `fg-canvas`。ゲームの世界そのもの。 |

> [!IMPORTANT]
> 「VIEW MAP」機能のように、上位レイヤー（5000）を透過・最小化させて下位（0）を表示させる場合、最前面の「戻る」ボタン（7000）が隠れないように注意してください。

## 2. UI 状態管理 (State Machine)

`UISystem.js` の `updateUI()` は、`game.state` に基づいて全 UI の表示・非表示を一括制御します。

### 2.1 `.hidden` クラスの責任範囲

`.hidden` クラスは、**ゲームの `state` 変数**に基づき、`updateUI()` が「要素を表示するか否か」を決定するための唯一の指標です。

- **原則**: 表示状態の制御に `style.display = 'none'` などのインラインスタイルを直接使用することは（たとえ確実性を期す目的であっても）**.hidden** クラスとの競合や予期せぬデグレードを招くため、**厳禁**とする。
- **付与**: `updateUI()` の冒頭で全 gameplay 要素に一括付与（決定論的リセット）
- **削除**: `switch(game.state)` 内の各 `case` が、その state で必要な要素の `hidden` を削除することで表示を制御
- **重要**: `updateUI()` 内で `.minimized` の有無を条件にした `hidden` 操作の分岐は行ってはならない。`.hidden` は state だけに従う

### 2.2 `.minimized` クラスの責任範囲

`.minimized` クラスは、**ユーザーの明示的な操作（「VIEW MAP」ボタンのクリック）**によってのみ付与・削除されます。

- **付与の契機**: ユーザーが「VIEW MAP」ボタンをクリックした時（`EventSystem.js` 内のリスナー）
- **削除の契機**:
  1. ユーザーが「BACK TO RESULT」ボタンをクリックした時（`EventSystem.js` 内のリスナー）
  2. `showResult()` が呼ばれた時（新しいミッションのリザルト表示開始時の初期化）
- **`updateUI()` は `.minimized` を操作しない**: `updateUI()` は `.minimized` の付与も削除も行ってはならない。`.minimized` はユーザー操作の状態であり、システムの状態（`game.state`）とは独立している

### 2.3 2つのクラスの関係

- `.hidden` がついていれば要素は非表示（`display: none` 相当）
- `.minimized` は要素が**表示されている状態**での見た目を制御する（透明度・サイズなど）
- `.minimized` がついていても `.hidden` がついていれば見えない。つまり**`.hidden` が最優先**

### 2.4 リザルトオーバーレイの正しいライフサイクル

| タイミング | `.hidden` | `.minimized` |
|---|---|---|
| ミッション開始（building, aiming）| ✅ 付与（非表示） | なし |
| ミッション終了 → `showResult()` 呼び出し | ❌ 削除（表示） | ❌ 必ず削除（前回の残存を初期化） |
| VIEW MAP クリック | （そのまま）| ✅ 付与 |
| VIEW MAP 中に `updateUI()` が走る | ❌ 削除（表示を維持） | （`updateUI()` は操作しない） |
| BACK TO RESULT クリック | （そのまま）| ❌ 削除 |
| リザルトを閉じる（CONTINUE 等）| ✅ 付与 | （`showResult()` 次回呼び出し時に初期化） |

### 2.5 既存仕様の確認

- **状態のリセット**: `showResult()` が呼ばれる際は、必ず `.minimized` クラスを明示的に除去する必要があります。JS でクラスを追加・削除するだけでは、次にその画面が表示された際に意図しない状態が引き継がれる可能性があるためです。
- **ブロックスコープ**: `switch(game.state)` 内の各 `case` は、変数の再宣言（`const`）エラーを防ぐため、必ず `{}` で囲んで独立したブロックスコープを持たせてください。

### 2.6 UI 責任マトリクス (UI Responsibility Matrix) [憲法]

UI 要素の表示・非表示（`.hidden` クラスの操作）は、以下のマトリクスで定義された「管理者」のみが行う権限を持つ。管理外の要素への干渉は、副作用とデグレードの原因となるため、**厳禁**とする。

#### 責任マトリクス (Responsibility Matrix)

| UI 要素 ID | 管理者 (責任メソッド) | 表示条件 (管理者内部ルール) | 備考 |
| :--- | :--- | :--- | :--- |
| `title-screen` | `updateUI()` | `game.state === 'title'` | - |
| `mission-hud` | `updateUI()` | `game.state !== 'title'` | 基盤パネル |
| `build-overlay` | `updateUI()` | `'building'`, `'aiming'`, `'event'` | サイドバー |
| `launch-control`| `updateUI()` | `game.state === 'aiming'` | 発射ボタン |
| `event-overlay` | `updateUI()` | `game.state === 'event'` | ターミナルパネル |
| `result-overlay`| `showResult()` | (手動) `MissionSystem` からの呼び出し時 | 排他的管理 |
| `result-overlay`| `resetResultOverlay()` | (手動) 非表示化が必要なタイミング | 排他的管理 |
| `receipt-overlay`| `showTerminalReport()`| (手動) ゲームオーバー時 | 排他的管理 |

#### 責任境界の定義
- **`updateUI()`** は、上記マトリクスで「管理者」とされている要素のみを操作する。特に `result-overlay` や `receipt-overlay` の `hidden` クラスを勝手に付け外ししてはならない。
- **`showResult() / showTerminalReport()`** は、自身の担当オーバーレイのみを操作する。管理外の `build-overlay` 等の表示を復元/非表示化する目的で、これらの要素の `hidden` クラスに触れてはならない。
- **`_updateTitleUI()`** は、タイトル画面移行時のクリーンアップとして、全 UI 要素を非表示にする最終責任を負う。

### 2.7 パネルの開閉状態管理 (Panel Expansion Logic) [憲法]

ビルドパネル（`build-overlay`）の開閉状態は、ユーザー体験を損なわないよう以下のルールで管理する。

1.  **イベント開始時の自動最小化**: `event` ステートに突入した際、`updateUI` は一度だけ `.collapsed` クラスを付与する。その後、ユーザーが手動で展開（`.collapsed` 除去）した場合は、その状態を維持しなければならない（`updateUI` が毎フレーム閉じ直すことを禁止する）。
2.  **イベント中の操作制限**: `event` ステートにおいて、ビルドパネル内のアイテム（パーツ・ロケット等）を選択・操作することを禁止する。
3.  **セクター遷移時の自動展開**: `building` ステートに移行する際（新セクター開始時）、パネルは必ず「展開（開いた）」状態でなければならない。このリセットはステート遷移を担当するメソッドの責任とする。

### 2.7 描画同期による残像防止 (Flash of Old Content Protection)

ブラウザのレンダリング最適化（バッチ処理）により、`hidden` の脱着と内容更新が同一フレームで行われると、CSS アニメーションのリセットが失敗し、前回の内容が一瞬表示されることがあります。これを防ぐため、以下の手順を厳守します。

1.  **強制リフロー (Forced Reflow)**:
    要素を非表示（`.hidden` 付与）にした直後、または内容をクリアした直後に `element.offsetWidth` 等のプロパティにアクセスし、ブラウザに「現在の状態（隠れた状態）」を強制的に確定させます。
2.  **描画フレームの分離 (rAF)**:
    `hidden` を除去して要素を再表示する際、`requestAnimationFrame(() => { ... })` を使用し、内容の更新が完了した**次の描画フレーム**で可視化を行います。

> [!TIP]
> `UISystem.resetResultOverlay()` と `showResult()` はこのルールに従って実装されています。新しいオーバーレイを追加する際もこのパターンを継承してください。

## 3. 描画同期とアニメーション

CSS トランジション（スライドアップ、フェードインなど）を JavaScript からトリガーする際、以下のテクニックが使用されています。

### requestAnimationFrame の2重呼び出し
```javascript
requestAnimationFrame(() => {
    requestAnimationFrame(() => {
        element.classList.add('active');
    });
});
```
- **目的**: 要素が `display: none` や `hidden` から復帰した直後に CSS アニメーションを開始させるため。
- **理由**: ブラウザのレンダリングパイプラインにおいて、要素が表示状態として計算されるのを確実に待ってからクラスを付与しないと、アニメーションが省略（スキップ）されて「パッと表示される」不具合が発生するためです。

## 4. スタイリングの優先順位

特定のボタン（LAUNCH ボタンなど）の配色を上書きする場合、`!important` の使用は極力避け、**詳細度を高めるセレクタ**を使用してください。

- **推奨**: `.btn-neon.btn-grad-orange:hover`
- **非推奨**: `.btn-neon:hover { border-color: orange !important; }`

詳細度を上げることで、共通テーマ（水色）と特定テーマ（オレンジ）を安全に共存させることができます。

## 5. タイトル画面の特別制御

タイトル画面 (`#title-screen`) は、通常のゲームプレイ UI とは完全に異なる独立したレイヤーとして管理されています。

- **早期リターン**: `UISystem.updateUI()` は、`game.state === 'title'` の場合に専用メソッド `_updateTitleUI()` を呼び出し、即座に処理を終了 (`return`) させます。これにより、ミッション用の HUD 表示ロジック（`terminal-panel` 等を `remove('hidden')` する処理）がタイトル画面に干渉するのを防いでいます。
- **UI の強制初期化**: `_updateTitleUI()` は、タイトル画面に対して `z-index: 9999` を付与し、他のプレイ用要素（ゲームキャンバス以外）を一括で非表示にすることで、いかなるステートから戻った場合でもクリーンなメニュー画面を優先表示させる「決定論的な」役割を担っています。
- **ID の峻別**: 実装上、`title-overlay`（旧称）ではなく、共通 ID として **`title-screen`** を正しく参照・操作する必要があります。

## 6. 入力およびインタラクションロジック (Input Logic)

ゲームプレイ中（`building`, `aiming` 状態を含む）におけるマウス/タッチ操作は、ドラッグ開始地点の**ワールド座標**および入力キーの状態に基づいて挙動が分岐します。操作の継続性を保証するため、`mousedown` はキャンバスで検知し、その後の `mousemove` および `mouseup` は `window` 全体でリッスンします。これにより、ドラッグ中にマウスが UI パネル（ターミナル等）の上を通過しても、エイミングやパン、回転操作が中断されません。

- **パン (Panning)**:
  - **条件**: 
    - PC: `Shift` または `Ctrl` キーを押しながらのドラッグ。
    - スマートデバイス: 2本指でのドラッグ（ピンチ/パス）。
  - **挙動**: マウス移動に合わせて `cameraOffset` が更新され、表示領域を平行移動させます。
- **エイミング (Aiming)**: 
  - **条件**: パン操作に該当せず、かつドラッグ開始地点がステージ中心から `boundaryRadius`（境界円、通常 400px 程度）の内側である場合。
  - **挙動**: マウス移動に合わせて `ship.rotation` が更新され、射出方向を調整します。
- **マップ回転 (Rotation)**:
  - **条件**: パン操作・エイム操作に該当しない場合（`boundaryRadius` 外側での通常ドラッグ）。
  - **挙動**: ステージ中心点に対するマウスの角度変化に基づき `mapRotation` を更新し、マップ全体を回転させます。

> [!IMPORTANT]
> 判定には `Game.getWorldPos()` を通した座標を使用してください。ズームやカメラオフセットが適用された状態でも、ステージ上の絶対的な位置（星がある領域かどうか）で判定を行うためです。
