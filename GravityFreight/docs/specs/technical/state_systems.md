# システム状態およびデータ管理仕様 (State & Data Systems)

本ドキュメントは、ゲーム全体を司るステートマシンの遷移制約、ならびにアイテムおよび永続化データの取り決めを定義します。

## 1. ゲームステートのライフサイクル制約

`game.state` はゲームの進行フェーズとUIを直結させる変数です。遷移には以下の制約が伴います。

- **`title`**: アプリの初期状態。アーカイブ画面（`archive`）からの戻り先として機能する。
- **`preparing` (開始準備)**: セクター移動時のみ経由する状態。このステート内で新しいマップ生成（`initStage`）を一度だけ集約実行すること。
- **`finishing` (演出待機)**: 物理移動が停止し、スコア・アイテムが確定した状態での収束演出専用ステート。この間は新たな物理計算・アイテム判定を厳禁とする。UI上は `mission-hud` 等の飛行中UIを維持する。
- **`archive` (アーカイブ)**: 過去の記録やランキングを閲覧・管理する状態。`title` からの遷移、およびリプレイ終了後の戻り先として機能する。
- **`replaying` (リプレイ再生)**: 記録されたデータを基にフライトを再現する状態。このステートでは入力による操作は無効化され、物理演算のみが再現される。
- **同一マップの継続**: `result` から `building` (再出撃) へ遷移する場合は、マップを再生成せず、ロケットの位置や耐久力などの一時ステートのみをリロードすること。

## 2. スタックアイテムと耐久力の管理 (Take-Use-Merge)

インベントリ内におけるアイテムの重複管理（スタック）と、消耗品使用時のデータ混濁を防ぐための変更手順です。

- **同一性判定**: アイテムのスタックは「ID」「現在の耐久力 (`charges`)」「強化内容 (`enhancements`)」の全要素が一致する場合にのみ結合される。
- **データ更新時の原則（Take -> Use -> Merge）**: 発射台等を使用する場合は、インベントリにあるポインタを直接変更してはならない。
  1. `Take`: 指定スタックからクローンとして「数量1」を取得し、元のスタックから切り離す。
  2. `Use`: 取得したクローンの耐久力等を処理・デクリメントする。
  3. `Merge`: 耐久が残っている場合は `{ isNew: false }` オプションを指定し、インベントリに戻して再結合させる（再取得として二重カウントされるのを防ぐため）。

## 3. 永続化データ機能 (Persistence)

セッション（ブラウザのリロード）を跨いで保持すべきデータ（ランキング、ユーザー設定、観測状態など）のメタ仕様です。

- **ストレージ・アクセスの一元化**: `localStorage` への直接アクセスは原則禁止し、`StorageUtils` を通じて行う。これにより、プライベートモード等でのアクセス拒否や例外処理を共通化する。
- JSON形式で保存する際、必ず現在のアプリケーションバージョン（`version`）をルートプロパティに含める。
- データのロード時、この `version` が最新と一致しない、または構造が異なる場合は、互換性エラーを防ぐため内容を破棄（リセット）する措置を講じること。

## 4. 静的データの一元管理 (Data Centralization)

マジックナンバーや重複する定義子を排除し、保守性を高めるための制約です。

- **施設情報の共通マッピング**: 施設の名称、ID（イニシャル）、UIカラー、および報酬定義は、`Data.js` の `FACILITY_INFO` オブジェクトに集約する。
- **参照の掟**: システム各部（`MissionSystem`, `StorySystem`, `TutorialUI` 等）で施設を判定・参照する場合、ハードコードされた文字列ではなく必ず `FACILITY_INFO[key].id` 等を使用すること。定数の値を変更するだけで、システム全体の整合性が保たれる構造を維持する。
