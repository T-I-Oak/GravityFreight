# UI アーキテクチャおよび機能仕様 (UI Architecture)

本ドキュメントは、UI の重なり順、表示状態の管理責任、およびスタイリングに関する設計制約（メタ情報）を定義します。

## 1. UI レイヤー構造 (Z-index 定義)

UI の重なり順は `z-index` によって厳密に階層化されています。新規要素の追加時は以下の制約に従うこと。

| Z-index | 対象要素 / カテゴリ | 役割と制約 |
| :--- | :--- | :--- |
| 7000 | 最前面アクションボタン | `launch-control`, `.result-actions`。いかなる演出よりも前面に配置し、常に操作可能にする。 |
| 6000 | 演出用オーバーレイ | `receipt-overlay`, `#story-overlay`。リザルト画面の上に重なる一時的な演出用。 |
| 5000 | メインオーバーレイ | `result-overlay`, `event-overlay` など。ゲームプレイを遮断する主要な UI。 |
| 1000 | UI コントロールコンテナ | HUD よりも前面だが、オーバーレイよりは背面。 |
| 50-100 | 恒常 HUD / パネル | `mission-hud`, `build-overlay`。プレイ中常に表示される情報レイヤー。 |
| 0 | ゲームキャンバス | 物理演算の舞台。 |

- 制約: 上位レイヤー（5000等）を透過・最小化させて下位を表示させる機能（VIEW MAP等）を実装する場合、最前面のボタン（7000）が背面に隠れない構造を維持すること。

## 2. 状態管理の責任範囲 (State Classes)

表示・非表示の制御にはインラインスタイル（`style.display = 'none'`等）を原則禁止し、CSSクラスによって一元管理します。

### 2.1 `.hidden` クラス（システム状態）
- 役割: ゲームのシステム状態（`game.state`）に基づく「表示/非表示」の絶対評価。
- 制約: 付与・削除は `UISystem.updateUI()`内でのみ行い、手動操作はゲーム進行と表示の矛盾を生むため禁止する。

### 2.2 `.minimized` クラス（ユーザー操作状態）
- 役割: ユーザーの手動操作（「VIEW MAP」ボタン等）による「展開/縮小」状態の保持。
- 制約: `.hidden` とは独立して機能し、表示状態（`.hidden` が無い状態）の中での見た目を制御する。`.hidden` の方が優先される。

## 3. UI 責任マトリクス

UIコンポーネントへのDOM操作（`.hidden`等の着脱）は、以下の管轄に基づくこと。

- `UISystem.js`: `game.state` に基づく全体の表示制御（Orchestrator）。
- `HUDManager.js`: 恒常パネル（スコア、セクター等）内の動的更新。
- 各種個別クラス (`ResultScreen`, `TerminalReport` 等): 管轄するオーバーレイ内部の排他的管理。`UISystem` はこれらの内部構造には干渉しない。

## 4. スタイリング原則とカラー運用

### 4.1 CSSのモジュール化と名前空間
- 機能ごとに分割された CSS（`ui.css`, `animations.css` 等）は、各画面の「ルート ID」を親セレクタに持ち、画面間の干渉を排除すること。
- 特定のボタンを装飾する際は、`!important` を避け、ルートIDを含む詳細度の高いセレクタ（例：`#launch-control .btn-neon`）を使用する。

### 4.2 デザイン・カラーのメタ運用ルール
- 施設やアイテムのカテゴリカラーは、ソースベース（Data.js 等）にて変数管理されている。
- 使用制約: カテゴリカラーは、アイテムカードの枠線、ホバー時のグロー（外側光彩）、およびパーティクル演出にのみ適用する。
- 禁止事項: 視認性を損なうため、カラーをテキスト要素の `color` 自体には適用しないこと（テキストは共通のフォントカラーを使用する）。

## 5. 入力と状態の永続化 (Input & Persistence)

ユーザーの環境設定および観測状態は、セッションを跨いで一貫性を保つこと。

### 5.1 マップ表示状態の保存
- **ズーム倍率**: マップの拡大・縮小操作（ホイール/ピンチ）が発生した際、その時点の倍率を `StorageUtils` を通じて `gf_map_zoom` キーに保存する。
- **復元**: `Game.js` の初期化時、保存された値が存在すればそれを優先して適用する。これにより、プレイヤーが好む観測スケールが維持される。
- **制約**: UIレイアウトを破壊するほどの極端な値（最小・最大値外）が保存されている場合は、初期値へフォールバックさせる。
