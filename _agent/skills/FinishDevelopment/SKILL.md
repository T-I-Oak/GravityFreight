# FinishDevelopment スキル

このスキルは、開発作業が完了した際に、プロジェクトの状態を最新化し、品質を担保した上で変更を `master` ブランチへ安全に統合するために使用します。

## 【最重要：絶対遵守・安全管理ルール】

### 1. アトミック・コミットの徹底
**「1ファイル更新ごとに、必ず個別にコミットする」**
1. `git add [修正ファイル]` を単独で実行する。
2. 完了を確認後、`git commit -m "[説明]"` を単独で実行する。
**注意：`&&` による連結実行は、絶対に行わないでください。**

### 2. 推測作業の完全禁止 (No Speculation)
**「指示されていない作業をオーナーの判断を仰がずに進めない」**

---

## 実行手順

### 1. 仕様書と更新履歴の最新化 (Spec & History)
- **仕様の更新**: `docs/specs/` 配下の各仕様書に未反映の変更点がある場合は更新してください。
- **更新履歴の記録**: **「実装モード [IMPL]」の場合のみ**、`docs/update_history.md` に今回の変更内容を追記してください。※[SPEC] モードでは不要です。
- **原則チェック**: `docs/guidelines/coding_principles.md` に準拠しているかセルフチェックを行い、結果をオーナーに報告してください。

### 2. クリーンアップと最終同期 (Cleanup)
- **一時ファイルの削除**: `tmp_` で始まる一時ファイルをすべて削除してください。
- **開発用ファイルの整理**: `dev_implementation_plan.md` や `dev_refresh_handover.md` など、マージに含めない `dev_` ファイルを削除または移動してください。
- **バージョン同期**:
    - [IMPL] モードの場合、`package.json` のバージョンが最新（最終修正報告時のもの）であることを確認してください。

### 3. 最終承認の獲得 (Final Approval)
- **[STOP!] ゲートの設置**:
    - 作業内容の要約をオーナーに提示し、**オーナーの明示的な承認を得るまで、マージおよびプッシュは行わないでください。**

### 4. マージとプッシュ (Merge & Push)
- 承認が得られたら、以下の手順を順番に実行します。
    1. `git add .` および `git commit -m "..."` (未コミットがある場合)
    2. `git checkout master`
    3. `git merge [feature-branch]`
    4. `git push origin master`

## 注意事項
- **[最重要] アトミック・コミット**: 1ファイルごとに git add / commit を単独実行。
- **Windows対応**: コマンド連結 (`&&`) は禁止。
- **update_history.md**: 必ずこのフェーズの最後（承認直前）に更新すること。
