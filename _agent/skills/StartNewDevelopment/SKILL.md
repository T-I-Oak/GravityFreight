# StartNewDevelopment スキル

このスキルは、プロジェクトで新しい機能開発やバグ修正を開始する際に、一貫した手順（ルール確認、タスク選定、計画策定）を踏むために使用します。

## 【最重要：開発中の絶対ルール】

### 1. アトミック・コミットの徹底
**「1ファイル更新ごとに、必ず個別にコミットする」**
1. `git add [修正ファイル]` を単独で実行する。
2. 成功を確認後、`git commit -m "[説明]"` を単独で実行する。
**注意：`&&` による連結実行は、いかなる場合も厳禁です。**

### 2. 推測作業の完全禁止 (No Speculation)
**「明示的な指示がない作業、推測に基づく修正は一切行わない」**
- 指示範囲を越えた「先走った修正」は厳禁。
- 指示が曖昧、または指示に含まれないが必要と思われる修正がある場合は、必ず作業前にオーナーに確認し、承認を得ること。

## 実行手順

### 1. プロジェクトルールの再確認と README の最新化
以下のディレクトリ・ファイル群を読み込み、現在のプロジェクトにおける規約を最新の状態で把握してください。
- **核心原則 (`docs/V2_CORE_PHILOSOPHY.md`)**: 共通化・階層化・意味論の定義。
- **コーディング原理 (`docs/guidelines/coding_principles.md`)**: 普遍的な設計思想。
- **運用ルール (`docs/guidelines/local_process_rules.md`)**: **AI専用の絶対ルール。**
- **ファイル管理 (`docs/guidelines/file_management.md`)**: 接頭辞（tmp_, dev_, local_）の定義。

**ドキュメントの最新化**:
読み込んだ結果、`docs/README.md` のインデックスが実態と乖離している場合は、作業開始前に `docs/README.md` を更新してください。

### 2. 現状の理解と引き継ぎの確認
`docs/planning/backlog.md` および関連する仕様書を読み込み、現在の開発フェーズと進捗状況を正確に把握してください。
- **引き継ぎファイルの確認**: もし `docs/dev_refresh_handover.md` が存在する場合は、**最優先でその内容を読み込み**、前回のチャットからの継続事項（次に着手すべき No. など）を確認してください。
- **指示の確認**: オーナーからの具体的な作業指示を待ってください。指示があった場合は、その内容が既存のドキュメントやルールと整合しているかを確認してください。

### 3. 開発計画 (dev_implementation_plan.md) の作成
オーナーから具体的な指示があったら、詳細な開発計画を **`docs/dev_implementation_plan.md`** として作成してください（アーティファクトではなくファイルとして保存）。

**計画の必須項目**:
- **作業モードの宣言**: 冒頭に `[SPEC]`（仕様策定）または `[IMPL]`（実装）を明記する。
- **連番 (No.) の付与**: 各項目に必ず No. を振り、進捗管理の共通言語とする。
- **詳細な記述**: 「何をどうするのか」を詳細に記載する。
- **チェックボックス形式**: 各項目は `- [ ]` を用いる。
- **作業順の構成**: 依存関係に基づき、着手すべき順番に並べる。
- **ステップごとの承認ゲート**: 「No. ごとにオーナーの承認を得てから次に進む」旨を明記。
- **テスト計画**: [IMPL] モードの場合、TDD を実施する範囲を定義する。

## 重要な制約事項 (遵守必須)
- **[最重要] アトミック・コミット**: 1ファイルごとに git add / commit を単独実行。
- **推測の禁止**: 指示されたことのみを行う。
- **コマンド実行**: Windows環境のため `&&` 連結は厳禁。
- **ファイルサイズ**: 1ファイル500行以内を目指す。
- **命名規則**:
    - `tmp_`: 一時ファイル（タスク終了までに削除）
    - `dev_`: 開発期間維持（マージ時に整理・削除）
    - `local_`: オーナー管理資産（AIは編集・作成禁止）
