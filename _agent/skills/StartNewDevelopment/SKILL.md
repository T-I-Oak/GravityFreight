# StartNewDevelopment スキル

このスキルは、プロジェクトで新しい機能開発やバグ修正を開始する際に、一貫した手順（ルール確認、タスク選定、計画策定）を踏むために使用します。

## 実行手順

### 1. プロジェクトルールの再確認
`docs/rules/` ディレクトリ内のすべてのドキュメントを読み込み、現在のプロジェクトにおける以下の各ルールを最新の状態で把握してください。
- **コーディング規約 (`coding_rules.md`)**: CSS設計、JS実装原則、データ管理など。
- **Gitワークフロー (`git_workflow.md`)**: ブランチ命名規則、コミット単位、マージ手順など。
- **プロセスルール (`local_process_rules.md`)**: Windows環境の制約 (`&&`禁止)、一時ファイル命名、TDDの必須条件、[STOP!]ゲートの設置。
- **テスト方針 (`testing_policy.md`)**: テストの構成、期待される品質基準。

### 2. 次に着手すべき項目の特定と提案
`docs/planning/backlog.md` を読み込み、以下の基準で「次に着手すべき項目」を選定し、ユーザーに提案してください。
- `Next` や `vX.X.X` などで優先度が示されている項目。
- 現在のプロジェクトの進捗（他のバックログの完了状況）に照らして、論理的に次に必要なステップ。
- ユーザーからの明示的な優先度指定がある場合はそれに従う。

### 3. 開発計画 (Implementation Plan) の作成
ユーザーが提案に同意（または項目の指定）をしたら、詳細な開発計画を作成します。計画には必ず以下の項目を含めてください。
- **目的と要件**: なぜこの変更を行うのか。必要に応じて `docs/specs/` を参照。
- **影響範囲**: 作成・修正するファイルの一覧。
- **具体的な手順**: 実施する作業のステップバイステップ。
- **技術的判断**: 実装方針や使用するデータ構造、デザインの方向性。
- **Windows対応の明示**: PowerShell環境であることを考慮し、コマンド結合 (`&&`) を絶対に行わないことの再確認。
- **テスト計画**: どのファイルにどのようなテストコードを書くか (TDDの適用)。
- **[STOP!] 承認ゲート**: 「開発計画の提示後、ユーザーの承認を得るまで作業を停止する」旨を明記。

## 重要な制約事項 (遵守必須)
- **コマンド実行**: Windows環境のため、`&&` を使用したコマンド連結は厳禁。1コマンドずつ実行すること。
- **ファイルサイズ**: 1ファイル500行程度を目安とし、超える場合はリファクタリング（ファイル分割）を提案に含める。
- **命名規則**: 一時ファイルは `tmp_`、Git管理外のローカルメモなどは `local_` で始めること。
