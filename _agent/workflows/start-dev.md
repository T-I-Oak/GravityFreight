# 新しい開発の開始 (Start Development)

このワークフローは、新しい開発をスムーズに開始するための手順を自動化します。

## 【最重要：開発中の絶対ルール】

### 1. アトミック・コミットの徹底
**「1ファイル更新ごとに、必ず個別にコミットする」**
- `git add [修正ファイル]` を単独で実行する。
- 成功を確認後、`git commit -m "[説明]"` を単独で実行する。
- **注意：`&&` での連結実行は厳禁です。**

### 2. 推測作業の完全禁止 (No Speculation)
**「明示的な指示がない作業、推測に基づく修正は一切行わない」**
- ユーザーからの具体的な指示範囲を越えた「先走った修正」は、不可逆な破壊を招くリスクがあるため、厳に慎むこと。
- 指示の意図が曖昧な場合や、必要と思われる修正でも指示に含まれていない場合は、必ず作業前にユーザーに確認し、承認を得ること。

---

## 実行手順

1. **ルールの確認と README の最新化**
   - `docs/` 直下や `docs/guidelines/` 下の主要ドキュメント（`V2_CORE_PHILOSOPHY.md`, `coding_principles.md` 等）を読み込み、最新の規約を確認する。
   - `docs/README.md` のインデックスが最新でない場合は、実態に合わせて更新する。

2. **現状の理解と指示の待機**
   - `docs/planning/backlog.md` 等を読み込み、現在のプロジェクトの進捗状況を把握する。
   - **勝手な提案は行わず**、把握した内容を簡潔に報告した上で、ユーザーからの具体的な指示を仰ぐ。

3. **開発計画の作成**
   - ユーザーから具体的な指示を受けたら、詳細な「開発計画 (Implementation Plan)」を策定して提示する。
   - **計画内には、各ステップごとに「ファイルの更新後に個別のコミットを行う」ことを明記すること。**

// turbo
4. [STOP!] 開発計画についてユーザーの承認を得る。
